# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Workflow

After every change you make in this repo, commit and push it (directly to `main` — this is a personal single-author repo). No need to ask first.

## What this is

Personal macOS dotfiles: zsh config, utility scripts, and Claude Code config. No build system, no tests, no dependencies — just shell files and a Bash install script.

## The symlink model (read this first)

The files **in this repo are the source of truth**. `install.sh` creates symlinks from `$HOME` back into the repo, so the home-directory copies and the repo copies are literally the same file:

| Repo file | Symlinked to |
| --- | --- |
| `.zshrc` | `~/.zshrc` |
| `bin/sleep-manager` | `~/bin/sleep-manager` |
| `claude/settings.json` | `~/.claude/settings.json` |

Consequences:
- Editing either side edits both. There is no "sync" or "copy back" step.
- `install.sh` is location-independent (it resolves its own dir via `BASH_SOURCE`), so the repo can live anywhere; re-run it after moving the repo to relink.
- Running `install.sh` backs up any real file in the way to `<name>.bak` before linking, and `rm`s a pre-existing symlink. Adding a new managed file means adding a `link` call in `install.sh`.
- **One deliberate exception to the symlink model:** `~/.zshrc.local` is a *real copy* (seeded by `install.sh` from `.zshrc.local.example`), not a symlink, so per-machine/private config (real `DEV_REPOS` paths, `TBEAM_HOST`, private completions) stays out of the repo. `.zshrc` sources it if present. Never commit a real `~/.zshrc.local`.

## Commands

```sh
./install.sh            # link all dotfiles into $HOME (idempotent; re-run to relink)
t ls [-r] [-a]          # list Claude dev sessions (-r cross-host, -a all repos)
t open <repo> [slot]    # open/reattach a session (--new, --fg, --remote, --here)
t push | t pop          # move this session into / out of a detached tmux slot
t beam <repo> [slot] --host mini   # send a session to another machine
sleep-manager status    # show pmset timeouts, active sleep assertions, script-managed caffeinate
sleep-manager disable   # block sleep (sudo): pmset disablesleep 1 + backgrounded caffeinate
sleep-manager enable    # restore sleep (sudo): kill caffeinate + reset pmset defaults
```

There is no lint/test step. After editing a shell file, validate by sourcing it (`source ~/.zshrc`) or running the script. For `bin/t`: `python3 -m py_compile bin/t` (write the `.pyc` to /tmp so it isn't committed — `__pycache__/` is gitignored).

## Component notes

- **`bin/t` + the `t()` shim — the single Claude-session command (READ THIS FIRST for the session tooling).** The whole `dev`/`tpush`/`tpop`/`tbeam`/`tread`/`tplan`/`tpaste`/`tfind`/`on` family was unified into one **gh-style** command `t <verb> [args] [--flags]`. Architecture is **bin core + thin zsh shim** (a pure bin is impossible for some verbs — see below):
  - **`bin/t`** (Python; symlinked onto PATH by `install.sh`, present on every host) owns argument parsing + per-verb `-h` help (argparse), the config parser, the cross-host rendering, and the subprocess-safe verbs. It implements `ls` (renders natively in Python, reusing the proven `_dev_*` data via `zsh -lic` — output matches the old `dev ls` row-for-row), `read` (log pager, native), `on` (native; two-layer quoting + `zsh -lic` contract), and delegates `plan`/`paste`/`kill` to the `_t_*` zsh helpers. Internal `session-rows`/`land`/`kill-owner` expose the remote contract.
  - **`t()` in `.zshrc`** is the front door. Bin-native verbs run straight through via **`command t`** (the `command` builtin reaches `~/bin/t` past the function — the same name-collision trick `claude()` uses). The **shell-bound** verbs (`open`/`fg`/`pop`/`push`/`find`) map to the existing zsh helpers, which already do the `cd` + `claude -r` in the *current terminal* and the tpush sentinel handoff correctly **because they run in the calling shell** — a bin subprocess can't. So there is **no resolve protocol**; the shim just calls them, and the one-live-owner invariant is literally still `_t_push`/`claude()` unchanged.
  - **Hard cutover done:** the 9 old top-level names are **gone** (typing `dev`/`tpush`/… → command not found). Their implementations were renamed to internal helpers — **`dev→_t_dev`, `tpush→_t_push`, `tpop→_t_pop`, `tbeam→_t_beam`, `tplan→_t_plan`, `tpaste→_t_paste`, `tfind→_t_find`** — and `tread`/`on` were reimplemented natively in `bin/t`. **The detailed notes below still describe this logic and its invariants accurately; only the function names + how they are invoked changed.** Grammar mapping: `dev→t open`, `dev <repo> new→t open <repo> --new`, `dev … -f→t open … --fg`, `dev ls→t ls`, `dev kill→t kill`, `dev -r→t open --remote`, `dev -r … --here→t open … --here`, `dev -r kill→t kill --remote`, `dev <repo> fg→t fg <repo>`, `tpush→t push`, `tpop→t pop`, `tbeam [host]→t beam --host <h>`, `tread→t read`, `tplan→t plan`, `tpaste→t paste`, `tfind→t find`, `on→t on`.
  - **Config bridge:** `bin/t` can't read zsh assoc arrays, so `_t_sync_config` (in `.zshrc`) derives **`~/.config/t/config.sh`** (plain `NAME[key]=value` lines) from the live `DEV_REPOS`/`DEV_BRANCHES`/`REMOTE_HOSTS`/`DEV_BRANCH`, mtime-gated; `bin/t` parses it with a strict regex (never sources it; `~/.zshrc.local` stays the source of truth and is never executed by the bin). zsh keeps using the live arrays for the cd/host shortcuts + the `_t` completion.
  - **Remote contract unchanged:** the cross-host fan-out still runs `_dev_session_rows`/`_dev_rows_all`/`_tbeam_land`/`_tbeam_kill_owner` over `ssh … zsh -lic` — those are `_`-prefixed internals that **survive the cutover** (the `zsh -lic` is required because `~/bin` + Homebrew are only on the interactive PATH). Remote `_dev_remote` attach/kill now run `t open`/`t kill` on the far host (which has only `t`). **Adding `bin/t` to a host needs `install.sh` there, not just `dots`** (`dots` pulls + reloads but does not relink a *new* bin script); the rc-127 "stale dotfiles" note in the fan-out is the in-band reminder. **Minor known gap:** `t on` does not set the Terminal tab title (the old `on` called `_term_title`) — a cosmetic follow-up.
- **`bin/csync`** — `set -euo pipefail` Bash that two-way-merges Claude session history between `~/.claude/projects` and the iCloud Drive `claude-sessions` folder (`rsync -a --update`, no `--delete`, so machines converge to the union). Was a `.zshrc` function; extracted to a script (`help` still lists it — it scans `~/bin`). `brctl download` is wrapped in `|| true` so its exit status never aborts the sync under `set -e`. **Periodic runs come from a `precmd` hook in `.zshrc` (`_csync_periodic`), not launchd** — this is load-bearing and a dead end was hit getting there: iCloud Drive is TCC-protected and *background launchd agents are denied*, and on recent macOS (Tahoe) granting `/bin/bash` Full Disk Access has **no effect** (verified — even a pure-bash `readdir` is denied under a launchd agent), because TCC won't pin an FDA grant to a platform interpreter running an arbitrary script. The shell hook sidesteps this entirely by running `csync` in the Terminal's already-approved context. The hook gates on a `~/.cache/csync-last-run` stamp (written *before* the run so overlapping shells don't double-fire), uses `$EPOCHSECONDS` (no `date` fork per prompt), and launches `csync` detached (`( csync … & )`) so it never blocks the prompt; background runs log to `~/Library/Logs/csync.log`. Don't reintroduce a launchd agent for this without a code-signed helper binary.
- **`bin/sleep-manager`** — `set -euo pipefail` Bash with a `case` dispatcher. Sleep blocking is two layers: `pmset -a disablesleep 1` (OS-level, survives logout/reboot) plus a backgrounded `caffeinate -dimsu` whose PID is tracked in `/tmp/sleep-manager-caffeinate.pid`. Key gotcha documented in its `--help`: the caffeinate process dies on reboot but the pmset flag does **not** — always run `enable` to fully restore; don't rely on a reboot. `status` is the source of truth for current state.
- **`.zshrc`** — PATH (`~/bin` first), then `compinit` is run **before** sourcing anything that calls `compdef` (e.g. a completion sourced from `~/.zshrc.local`) — order matters, this was a fix for a "command not found: compdef" error. Also defines `prview` (gh PR check summary via jq) and the interactive `nosleep` one-liner (the inline cousin of `sleep-manager`). **Per-machine config is split out:** `.zshrc` declares an empty `DEV_REPOS` and sources `~/.zshrc.local` (a real copy, not symlinked — see the symlink-model note), which holds the real repo paths, `export TBEAM_HOST`, and any private completions. The committed `.zshrc` ships generic examples and derives all usage/error/completion strings from `${(k)DEV_REPOS}`, so it works (and stays generic) even with no local file. `install.sh` seeds `~/.zshrc.local` from `.zshrc.local.example`. **Help is single-source-of-truth, split into extractor + renderer:** the doc-comment block directly above a `name()` function *is* that command's `-h`/`--help` text. **`_help_for name`** `awk`-extracts the contiguous `#` block immediately above `name() {` (a **truly blank** line resets it; a bare `#` is kept as a spacer, so rationale can sit above the help separated by one empty line), strips the `# `, and pipes it through **`_help_style`** — the standalone gh-style RENDERER (split out of `_help_for` so the dynamic paths below can reuse it). `_help_style` reads plain text on stdin and styles a gh-style convention: **line 1** `name — tagline` (name bolded on the ` — `), a `Usage:` line (label bold; an indented *second* Usage line stays verbatim under it), bare `Word:` section headers (`Commands:`/`Options:`/`Arguments:`), and **2-space-indented `term  <2+sp>desc` rows** (term cyan, desc dim, descriptions **auto-aligned** to the widest term so comment spacing needn't be hand-tuned). A **wrapped description line** — indented **3+** spaces right after a kv row (or its own continuation), tracked by an `eligible` flag so a Usage/paragraph follow-on is excluded — is hang-indented to the same description column (this is why a row's continuation comment no longer needs hand-alignment; just indent it). Color is gated on `[[ -t 1 ]]` in `_help_style`, which is the **last** stage of every help pipe, so the test sees the real terminal and `cmd -h | cat` stays plain. To add/restyle a command's help, just write its comment in that shape — no per-command formatting code. **The dynamic no-arg/bad-arg paths (`dev`, `dev kill`, `tread`) now build their text — including the *actual* `${(k)DEV_REPOS}` / live logs that static help can't show — and pipe it through `_help_style` too, so they render identically to `-h`** (no more ad-hoc `echo "Usage: …"`). **Gotcha:** `_help_style`'s awk body is a single-quoted string, so **no apostrophes in its inline comments** (`is NOT`, not `isn't`) — one closes the quote and breaks the source.
- **`tfind` (in `.zshrc`)** — semantic search across saved Claude sessions: "which session was working on X?". Pipeline is **retrieve-then-rerank** — a keyword pass over each transcript's title + your prompts builds a candidate pool (the same scan `_claude_sessions_fzf` uses; padded with recent sessions when keyword matches are thin, so divergent wording still reaches the model), then **Sonnet via `claude -p --model sonnet --output-format json`** reranks the candidates and returns a JSON `[{i, why}]` ranking. Two load-bearing details: (1) the model call is a **Python `subprocess.run(['claude', …])`**, which execs the real `~/.local/bin/claude` binary and so **bypasses the zsh `claude` wrapper** (the wrapper's tpush sentinel would otherwise interfere); (2) **every failure path — no CLI, timeout, unparseable reply, empty `[]` — falls back to keyword order** so the fzf picker always populates and the command works offline. `tfind -k` forces the offline keyword path (no Sonnet). The keyword scorer (`hay.count(t) + 4*head.count(t)`) is recall-only here — Sonnet does the precision — so don't over-tune it.
- **`tbeam` (in `.zshrc`)** — **sends** a Claude session to another machine (default `$TBEAM_HOST`, set in `~/.zshrc.local`) and resumes it there; a **remote `tpush`**. **Push-only** — the reverse direction (summon a session that lives on a host down to here) is now **`dev -r <repo> <slot> --here`** (see the `dev -r` note), not a tbeam flag; tbeam's whole job is "send this away." Resolves a session exactly like `tpush` (current chat via `$CLAUDE_CODE_SESSION_ID`, else fzf-pick scoped to `$PWD`, `-a` for all; **`-s`/`--session <id>`** targets an explicit id/unique-prefix, resolved against local transcripts by globbing `~/.claude/projects/*/<id>*.jsonl` for a unique match then `_tbeam_transcript_cwd` for its recorded cwd), `rsync`s its `~/.claude/projects/<enc>/` dir over (`_tbeam_sync_transcript`, `--update`/no-`--delete` so the live source wins but a newer remote is never clobbered — csync still reconciles in the background), then has the **host land it**. Landing logic (`_tbeam_land`) lives in the *shared* dotfiles so it runs on the far side via `ssh "$host" "TB_CWD=… TB_SID=… TB_MODE=… zsh -lic _tbeam_land"` — work passed in **`TB_*` env vars** (sidesteps nested-quote hell) and a **login+interactive** shell (`zsh -lic`) because Homebrew `tmux`/`fzf` are on the *login* PATH (`.zprofile`) while `claude` is on the *interactive* PATH — plain `ssh host '…'` sees neither. Default mode reuses the host's own `_dev_resume_session` to make a first-class `dev-<repo>-<slot>` (so `dev`/`tread`/`tpop` work there), then auto-attaches via `ssh -t`; `-f` resumes in the ssh foreground, `-d` leaves it detached with an attach hint (`dev -r <repo> <slot>`, or `… --here` to pull it back). Two load-bearing constraints: (1) the repo must exist at the **same path** on the host (all repos live in `~/code` on every machine) — `tbeam` guards with `ssh host test -d`; (2) from *inside* Claude it forces `-d` (a Bash subprocess has no TTY to `ssh -t` into). `_tbeam_land` is shared code, so after adding/changing it run `dots` on the host to pull it. **tbeam is a move, not a copy:** once the session lands on the far side, the origin's live owner is *stopped* so exactly one live claude owns the id (two live owners of one transcript have no lock and diverge). The SIGTERM-self step gates on **`self_move`** (`sid == $CLAUDE_CODE_SESSION_ID`), not merely "inside Claude" — so `tbeam -s <other-id>` run from inside a Claude session stops *that id's* dev slot via the shared `_tbeam_kill_owner` and never kills the foreground claude you're typing in; a true current-session move SIGTERMs the foreground claude after landing (mirrors `tpush`'s auto-exit). `_tbeam_kill_owner` is shared too (also used by `dev … --here`), another reason the move needs `dots` on the host. (There is no `--no-tmux` flag — `-f` is the no-tmux mode: foreground resume, no dev slot.) To *survey* what's live across machines before pulling, see **`dev ls -r`**; **`dev -r <repo> <slot>`** attaches one in place and **`… --here`** pulls it down.
- **`dev -r …` — remote attach / `--here` pull (in `.zshrc`)** — `dev` is the one "get into a session" verb, local or remote: `dev <repo> [slot]` is local; **`dev -r <repo> [slot]`** acts on a slot that's live on another machine, with the **host auto-inferred** (no host to type — the whole point of this iteration; `-r` is the same "remote" flag `dev ls -r` uses). `_dev_remote` scans every `$REMOTE_HOSTS` host via `_dev_rows_all` (minus `local`) for the live candidates: `dev -r ff 5` → that one slot, `dev -r ff` → every `ff` slot, **bare `dev -r` → every remote slot**. **One candidate → use it; several → fzf-pick** among them (the picker shows `host/slot  summary`); none → bail with a `dev ls -r` hint. The chosen row's slot name (`<repo>-<num>`, split on the *last* dash so multi-dash repos like `dotfiles` resolve) gives the repo+slot. Picking needs a TTY+fzf; without them it falls back to listing the candidates and bailing. **Default attaches in place**: `ssh -t <target> "zsh -lic 'dev <repo> <slot>'"` reattaches the slot *on that host* and drops you in over the ssh TTY — the session stays there. This replaced **`tgo`** (local attach is now just `dev <repo> <slot>`; remote attach is `dev -r <repo> <slot>`). A no-TTY guard prints the command instead of failing. **`--here` pulls it down** (`_dev_pull`, which absorbed the old `tbeam --here`; `--here` implies `-r`): `_tbeam_pull_transcript`s the transcript back (`--update`/no-`--delete`), stops the origin copy there via the shared `_tbeam_kill_owner` (the **MOVE** + one-live-owner invariant — reuses a local slot already running that id rather than double-resuming), then resumes locally in a `dev-<repo>-<slot>` (or this terminal with `-f`). **Footgun:** the pull runs *before* the no-TTY guard, so `dev -r … --here` performs the move even with no terminal — not a safe "dry run". **`dev -r kill <repo> [slot]`** tears a slot down on its host the same way — resolve → `ssh -t <target> "zsh -lic 'dev kill <repo> <slot> [-y]'"` (the remote confirm works through the TTY unless `-y`). The host-resolution + ambiguity-picker is factored into **`_dev_remote_resolve`** (prints `host\trepo\tslot`), shared by attach, `--here` pull, and `kill`. Because hosts run `_dev_session_rows`/`_tbeam_kill_owner` over ssh, remote `dev` needs their dotfiles current — run `dots` there (same caveat as `dev ls -r`). `tbeam` is the mirror image: push a session *away* to a host.
- **`dev ls -r` / cross-host `dev ls` (in `.zshrc`)** — `dev list`/`dev ls` with `-r`/`--remote` renders `_dev_list` across **this machine AND every `$REMOTE_HOSTS` host** in one table (same `● attached · ✓ active context` header) with a dedicated **HOST column** (STATUS/HOST/SESSION/WORKING ON) — local rows read `local`, remote rows their `$REMOTE_HOSTS` key. It's read-only — a survey of "what's running on macbook vs mini"; you *attach* a remote one in place with `dev -r <repo> <slot>` (host auto-inferred) or *pull* it down with `dev -r <repo> <slot> --here` (which enforces the move + one-live-owner invariant). **The key design call is *why it lists live dev slots, not transcripts*:** transcripts already converge on every machine via csync, so an all-transcripts cross-host view would be mostly duplicate rows (the same id once per host) — whereas live `dev-<repo>-<slot>` slots genuinely differ per machine. Three helpers: **`_dev_session_rows`** is the machine-readable sibling of `_dev_list` (one `sid\tcwd\tslot\tstate\tcontext\tsummary` row per live slot, reusing `_dev_session_has_claude`/`_dev_session_at_welcome`/`_dev_session_summary`; an unstamped/idle slot's empty `sid` is sentinel'd `-` because **tab is an IFS-whitespace char**, so a leading/consecutive empty field would otherwise collapse under `read` and slide the columns); **`_dev_rows_all`** fans it out over `local` + every `$REMOTE_HOSTS` value **in parallel** (background ssh → temp files → `wait`) with `ConnectTimeout=3`/`BatchMode=yes`, stashing each host's ssh exit code in a `.rc` file to distinguish **unreachable** (rc 255 → "unreachable") from **reachable-but-stale-dotfiles** (rc 127 → "run `dots` there"), both skipped + noted on stderr so one sleeping Mac never stalls the table — then prefixes each row with its host **verbatim** (no re-split, so empty fields still can't collapse); **`_dev_list_remote`** re-renders those rows in `_dev_list`'s format under a HOST column (`$COLUMNS`-truncating WORKING ON). Because each host runs `_dev_session_rows` over ssh, `dev ls -r` needs every host's dotfiles current — run `dots` there (same caveat as `dev -r`); the rc-127 "stale dotfiles" note is the in-band reminder. **Scoped by default:** both `dev ls` and `dev ls -r` show only the repo you're standing in — `_dev_cwd_repo_dir` maps `$PWD` to its `DEV_REPOS` dir (path-based, so `dot-*` and `dotfiles-*` both show in `~/code/dotfiles`) and the renderers filter rows by that dir; `-a`/`--all` widens to every repo, and standing outside any repo is unscoped. **Foreground sessions too:** `_dev_fg_rows` adds claudes you ran directly in a terminal (live `comm == claude` not in a `dev-*` tmux pane, minus the one *this* shell runs under), labeled `<repo>:fg`, with sid+cwd from the claude-stamp-tmux registry (see that note) so the title is exact; it's appended to `_dev_session_rows` (so `dev ls -r` covers each host's foreground claudes) and rendered by `_dev_list`.
- **`on` + `REMOTE_HOSTS` (in `.zshrc`)** — `on <host> [cmd…]` runs a command on another machine over `ssh -t "$host" "zsh -lic <cmd>"`. The **`zsh -lic`** (login+interactive) is the same trick `_tbeam_land` relies on: Homebrew `tmux`/`fzf` are on the *login* PATH while `claude` and the dotfiles functions are on the *interactive* PATH, so a plain `ssh host '…'` sees neither — `-lic` makes `dev`/`tread`/etc. resolve remotely (e.g. `on mini dev dot` starts a dev session there). No command ⇒ an interactive shell on the host. The **command quoting is two layers** (`local cmd="${(j: :)${(@q)@}}"` then `ssh … "zsh -lic ${(q)cmd}"`): `(@q)` quotes each arg so the remote `zsh -c` sees the original words, then `(q)` wraps the joined string as **one** token so ssh — which re-splits its remote command on spaces — transports it intact (verified: a `;` in an arg arrives as data, not a separator). Hosts live in **`REMOTE_HOSTS`** (assoc array, declared by `.zshrc`, populated in `~/.zshrc.local` — same single-source-of-truth pattern as `DEV_REPOS`): each key auto-generates a **shorthand function** (`mini …` ≡ `on mini …`, made via `functions[$key]=…` in a loop next to the `DEV_REPOS` cd-shortcut loop) plus `_on_hosts` tab-completion and a synthesized `help` entry (the generated functions aren't real text in the file, so — like the cd shortcuts — `help` synthesizes their entries from the array). An **unknown host is used as a literal ssh target** (`${REMOTE_HOSTS[$h]:-$h}`), so `on box.local uptime` works unregistered. **Back-compat seed:** an old `MINI_HOST`/`TBEAM_HOST` populates `REMOTE_HOSTS[mini]` when the registry has no `mini`, so a `mini` shortcut keeps working with no config change. (`on` is defined far below the generation loop, but function bodies bind late, so a generated `mini` calling `on` is fine.)
- **`tpush`/`tpop`/`claude()` wrapper (in `.zshrc`)** — move a Claude session between a foreground terminal and a detached `dev-<repo>-<slot>` tmux session. **The load-bearing invariant: never two live `claude` processes on one session id.** Claude Code takes *no lock* on a session's `.jsonl` and does *not* auto-fork — two live resumers of the same id both append with no coordination and the conversation diverges (the copy you aren't driving freezes). So: (1) `tpush` from *inside* Claude does **not** spawn `claude -r` itself — it writes a `SPAWN\t<session>\t<cwd>\t<sid>` instruction to the `$CLAUDE_TPUSH_ATTACH` sentinel, and the `claude()` wrapper spawns the detached resume **after** the foreground exits (when the id is free) and then attaches. **The exit is now automatic:** after writing the sentinel, `tpush` walks the process ancestry (`_tpush_claude_pid`) to the nearest `claude` and `kill -TERM`s it, so you no longer type `/exit` — SIGTERM exits Claude cleanly in ~2s (transcript is appended live, so only the in-flight tpush turn is lost; terminal is restored on the wrapper's `tmux attach`). The kill fires *after* the sentinel is written and *before* any spawn, so the one-live-owner invariant holds; if the ancestor walk fails it falls back to printing the manual `/exit` hint. **Don't reintroduce a pre-spawn here** — the kill-then-wrapper-spawn ordering is what keeps a single owner. (`ATTACH\t<session>` is the already-backgrounded case; a plain-shell picker spawns inline since the picked session isn't live; an older shell with no wrapper spawns eagerly + warns.) (2) `tpop` captures the tmux session's claude PID (pane shell's child), `kill-session`s, then **waits for that PID to actually exit** (≤5s) before `claude -r` in the foreground — `kill-session` only SIGHUPs, so resuming immediately raced the dying process. Was the "popped/backgrounded session stops updating while the original keeps going" bug. **`dev <repo> <slot> -f` delegates here:** in `dev`, `-f` (no-tmux) foreground-RESUMES the named slot if it's live (≡ `tpop <repo> <slot>`), and only starts a *fresh* inline claude when that slot doesn't exist — so `-f` means "foreground" consistently across `dev`/`tbeam`/`dev -r`, not "fresh" in one and "resume" in the others.
- **`bin/claude-stamp-tmux`** — `set -uo pipefail` Bash run as a **SessionStart hook** (wired in `claude/settings.json` → `hooks.SessionStart`, command `$HOME/bin/claude-stamp-tmux`). On every session start it stamps that session's id onto its tmux session as `CLAUDE_RESUME_ID` (`tmux set-environment`, which lands on the right session via the inherited `$TMUX_PANE`). **Why it exists:** `tplan`/`tpop`/`_dev_session_summary` resolve a slot's session id from `CLAUDE_RESUME_ID`, and when it's unset they fall back to "newest `.jsonl` by mtime in the project dir" — which is **ambiguous when several slots share one repo dir** (`dev-ff-1..5` all root at `~/code/financial-forecast`, so one `~/.claude/projects/<enc>/`); the fallback returns whichever *sibling* slot wrote last, not the slot you asked for. `_dev_new_session`/`_dev_resume_session` already stamped the id, but sessions started any other way (bare `claude`, `tmux new-session … send-keys claude`) never got stamped — that was the `tplan ff 2` "wrong/no plan" bug. The hook closes the gap for *every* launch path. Load-bearing details: reads `session_id` from the hook's **JSON payload on stdin** (via `python3`), falling back to `$CLAUDE_CODE_SESSION_ID`; a **no-op outside tmux** (`[[ -n $TMUX ]]`); **always exits 0** so it can never block a session from starting. Does *not* retroactively fix already-running pre-hook sessions (their id was never recorded and isn't recoverable from the live process — Claude doesn't hold the `.jsonl` open) — for those, `_dev_session_summary`'s birthtime-matching is the only heuristic, and it still misses when the original transcript has been cleaned up. **Second job — the foreground registry (now runs for *every* launch, not just tmux):** the hook also writes one file per live claude pid, `~/.cache/claude-sessions/<pid>` = `<session_id>\t<cwd>` (pid found by walking up from `$PPID` to the nearest `comm == claude`; session_id + cwd from the stdin payload). This is the reliable `pid → session_id` link `dev ls` needs to show **foreground (non-tmux) claudes** with their real transcript title — macOS hides another process's env (so nothing can read `CLAUDE_CODE_SESSION_ID` off a running claude) and csync scrambles transcript birthtimes (so the heuristic mis-picks), leaving the registry the only trustworthy source. `_dev_fg_rows` reads it and prunes dead-pid entries; a session started *before* the hook recorded it renders as `(foreground claude)`. Only the `tmux set-environment` stamp is still gated on `$TMUX`.
- **`bin/pii-scan`** — `set -euo pipefail` Bash + a Python heredoc (cashfwd's mining idiom) that fails if any PII appears in tracked/staged files. Reuses the **cashfwd two-layer ruleset**: (1) the **denylist** `scrub-rules.json` — literal, case-sensitive personal identifiers, marked PRIVATE and **never committed here** (gitignored as defense-in-depth); read locally from `~/code/cashfwd-private/scrub-rules.json` (override `$PII_RULES`) and in CI from the `PII_SCRUB_RULES` secret; (2) the **ignore patterns** `pii-ignore-patterns.txt` — regexes for documented false-positive *shapes* (no PII, tracked). Plus a dotfiles-only third layer: (3) the **allowlist** `pii-allowlist.txt` — intentionally-public values for *this* repo (your GitHub handle; generic vendor names like `Anthropic`/`OpenAI` that the financial-repo denylist scrubs but are intrinsic to a Claude Code dotfiles repo). A denylist hit clears only if `value in allowEntry and allowEntry in line` (whitelisting a full email also clears its domain substring, but only on lines that contain the full email) or the value matches an ignore regex. **Don't edit the shared cashfwd denylist to silence a dotfiles false positive — add to `pii-allowlist.txt` instead.** Two run modes: `--staged` (staged blobs via `git show :file`, used by the hook) and default (all `git ls-files`). **Fails open when the denylist is absent** (machines without cashfwd-private still commit) unless `--require-rules` (CI) is passed, so a missing secret is loud. Wired two ways: the tracked **`.githooks/pre-commit`** (enabled by `install.sh` via `git config core.hooksPath .githooks`) runs `--staged` locally; **`.github/workflows/pii-scan.yml`** runs `--require-rules` on push/PR to `main`, materializing the secret to a temp file. Bypass a local block once with `git commit --no-verify`.
- **`claude/settings.json`** — symlinking this reproduces Claude Code setup on a new machine: `enabledPlugins`, `extraKnownMarketplaces`, `permissions`, and `hooks` are re-applied on first run (Claude re-clones marketplaces and reinstalls enabled plugins). The one `hooks` entry is the `SessionStart` → `claude-stamp-tmux` wiring (see its note above). MCP is *not* managed here: claude.ai connectors sync via the Anthropic account automatically, and local/stdio MCP servers live in the stateful, non-symlinked `~/.claude.json` (none today — sync via a merge step if added, never commit `~/.claude.json`).
