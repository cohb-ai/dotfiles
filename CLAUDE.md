# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Workflow

`main` is protected ŌĆö it takes changes through PRs, not direct pushes. **Each new `t open <repo>` session now opens in its own git worktree on a dedicated branch (`dev/<basename>-<slot>`) freshly branched off `origin/main`** ŌĆö the worktree-per-session model (see the `_dev_worktree_*` note below), which replaced the old single shared `dev/claude-1` tree where sibling sessions trampled each other. Commit and push to that per-session branch as you go ŌĆö no need to ask first ŌĆö and open a PR to `main` when a change is ready; once it merges, a periodic sweep removes the worktree + branch automatically. `dev/claude-1` survives only as the dev clone's parking branch and the fallback `$DEV_BRANCH` for opt-out repos (`DEV_WORKTREE[repo]=0`). **Editing a session's worktree is NOT live** (the `$HOME` symlinks point at the main worktree, never a session worktree) ŌĆö landing a change means mergeŌåÆ`dots`, exactly as before.

## What this is

Personal macOS dotfiles: zsh config, utility scripts, and Claude Code config. No build system and no runtime dependencies ŌĆö just shell files and a Bash install script. The only tests are a `pytest` suite for the pure logic in the Python bins (`bin/t`, `bin/pr-watch`), with CI-only dev deps; `install.sh` stays dependency-free.

## The worktree-separation model (read this first)

There are **two checkouts of this repo on every machine**, and they have different jobs:

| Checkout | Path | Branch | Role |
| --- | --- | --- | --- |
| **dev clone** | `~/code/dotfiles` | `dev/claude-1` | where you + agents develop; `DEV_REPOS[dotfiles]` / `t open dotfiles` point here |
| **main worktree** | `~/.local/share/dotfiles-main` (override `$DOTFILES_MAIN_WT`) | `main` | the **LIVE surface** ŌĆö `$HOME` symlinks point here |

Both are linked worktrees of one shared `.git`. **`install.sh` creates the main worktree and points the `$HOME` symlinks at it; `dots` keeps it fast-forwarded to `origin/main`.** The dev clone stays on `dev/claude-1` as the base from which sessions branch.

**A third layer sits on top of this:** `t open` no longer develops *in* the dev clone ŌĆö each new dev session opens its own **per-session worktree** (`~/code/.worktrees/<repo>/<slot>`, on `dev/<basename>-<slot>` off `origin/main`), reaped once merged. See the `_dev_worktree_*` note below. The dev clone is just `DEV_REPOS[dotfiles]` ŌĆö the repo those worktrees are `git worktree add`'d from ŌĆö and `dev/claude-1` is now only its parking branch.

**Why two trees (this is the whole point):** the old single-tree model symlinked `$HOME` straight at the dev clone, so `dots` (which did `git checkout main`) swapped the live files out from under whatever branch/work was in that one shared tree ŌĆö and because the tree is shared by every shell/agent, one `dots` (or a stray `git checkout main`, e.g. a concurrent agent's) reverted everyone's live config and un-applied in-progress work. Splitting them fixes it structurally: **git physically refuses to check out `main` in the dev clone while the worktree holds it**, so the live surface can only move via `dots`, and `dots` never touches the dev clone. You can develop and pull released updates in any order without collisions.

**`dots` (in `.zshrc`) ŌĆö two modes:**
- **`dots`** (default) ŌåÆ fast-forward the main worktree to `origin/main` + reload. Live = released `main`. Never touches the dev clone, never switches a branch. On an old single-tree machine the first run **migrates in place** (moves the clone off `main` to `dev/claude-1`, creates the worktree, repoints the symlinks ŌĆö aggressive by design). It resolves the live surface from the `~/.zshrc` symlink and the dev clone from the shared `--git-common-dir`, so it works wherever the repo was cloned.
- **`dots --dev`** ŌåÆ flip the live symlinks to the **dev clone** (current branch) so in-progress edits are instant-live for testing; a later plain `dots` flips back to the worktree. This is the only way dev edits go live (under the new model, editing a repo file is *not* automatically live unless the symlinks point at that tree ŌĆö see below).

**`install.sh` link source** is chosen by `DOTFILES_LINK_DEV`: unset ŌåÆ `LINK_SRC` = the main worktree (default, live = released main); `DOTFILES_LINK_DEV=1` ŌåÆ `LINK_SRC` = the invoking dev clone (set by `dots --dev`). `setup_main_worktree()` does the migrate + worktree create/FF; it probes `$MAIN_WT/.git` to decide existence (robust to `/tmp`ŌåÆ`/private/tmp` path-canonicalization that defeats matching `git worktree list` output).

**Migration / rollout:** because the *new* `dots`/`install.sh` only exist after the change is itself live, bootstrap each machine once by running `./install.sh` from `~/code/dotfiles` (or the old `dots`, then `./install.sh`); after that the new `dots` maintains the worktree. Adding a host still needs `install.sh` there (not just `dots`).

### The symlink model (the older half, still true)

The files **in the live surface are the source of truth** for what runs. `install.sh` creates symlinks from `$HOME` into that checkout, so the home-directory copies and the checkout copies are literally the same file:

| Repo file | Symlinked to |
| --- | --- |
| `.zshrc` | `~/.zshrc` |
| `bin/sleep-manager` | `~/bin/sleep-manager` |

Consequences:
- Editing the **live checkout** (whichever tree the symlinks currently point at ŌĆö the main worktree by default, or the dev clone after `dots --dev`) edits the `$HOME` copy too; there is no "sync" or "copy back" step. **But editing the dev clone while the symlinks point at the worktree is NOT live** ŌĆö that's the deliberate separation; run `dots --dev` to make the dev clone live, or merge to `main` + `dots`.
- `install.sh` is location-independent (it resolves its own dir via `BASH_SOURCE`), so the repo can live anywhere; re-run it after moving the repo to relink.
- Running `install.sh` backs up any real file in the way to `<name>.bak` before linking, and `rm`s a pre-existing symlink. Adding a new managed file means adding a `link` call in `install.sh`.
- **Deliberate exceptions to the symlink model:** `~/.zshrc.local` is a *real copy* (seeded by `install.sh` from `.zshrc.local.example`), not a symlink, so per-machine/private config (real `DEV_REPOS` paths, `TBEAM_HOST`, private completions) stays out of the repo. `.zshrc` sources it if present. Never commit a real `~/.zshrc.local`. **`~/.claude/settings.json`** is the same pattern ŌĆö a real copy seeded from `claude/settings.json.example` (hook only), *always*, because Claude Code **writes** to the live file at runtime (`/model` saves the default model, "always allow" appends permission rules, plugin toggles land there); a tracked/symlinked copy kept the repo dirty and risked committing private allow-rules, so the old author-mode symlink (install choice 2 / `CLAUDE_SETTINGS=author`) was dropped ŌĆö `install.sh` materializes a legacy symlink into a real copy of its current content. An existing settings file is never clobbered, and `claude/settings.json` is gitignored as defense-in-depth.

## Commands

```sh
./install.sh            # link all dotfiles into $HOME (idempotent; re-run to relink)
t ls [-r] [-a]          # list Claude dev sessions (-r cross-host, -a all repos)
t open [repo] [slot]    # open/reattach (repo defaults to cwd repo; --new, --fg, --remote); auto-attaches a remote-only slot in place
t beam [repo] [slot] --from <host>   # pull a session from another machine to here (receive); omit --from to send
t push | t pop          # move this session into / out of a detached tmux slot
t beam <repo> [slot] --host mini   # send a session to another machine
sleep-manager status    # show pmset timeouts, active sleep assertions, script-managed caffeinate
sleep-manager disable   # block sleep (sudo): pmset disablesleep 1 + backgrounded caffeinate
sleep-manager enable    # restore sleep (sudo): kill caffeinate + reset pmset defaults
```

After editing a shell file, validate by sourcing it (`source ~/.zshrc`) or running the script. For `bin/t`: `python3 -m py_compile bin/t` (write the `.pyc` to /tmp so it isn't committed ŌĆö `__pycache__/` is gitignored).

For the Python bins there is a test suite: `pip install -r requirements-dev.txt` once, then `python3 -m pytest` (add `--cov --cov-report=term-missing` for coverage). CI runs it as the `pytest` job, gated by `--cov-fail-under` ŌĆö see the `tests/` note below.

## Component notes

- **`bin/t` + the `t()` shim ŌĆö the single Claude-session command (READ THIS FIRST for the session tooling).** The whole `dev`/`tpush`/`tpop`/`tbeam`/`tread`/`tplan`/`tpaste`/`tfind`/`on` family was unified into one **gh-style** command `t <verb> [args] [--flags]`. Architecture is **bin core + thin zsh shim** (a pure bin is impossible for some verbs ŌĆö see below):
  - **`bin/t`** (Python; symlinked onto PATH by `install.sh`, present on every host) owns argument parsing + per-verb `-h` help (argparse), the config parser, the cross-host rendering, and the subprocess-safe verbs. It implements `ls` (renders natively in Python, reusing the proven `_dev_*` data via `zsh -lic` ŌĆö output matches the old `dev ls` row-for-row), `read` (log pager, native; `-d`/`--dump` + `-n N`/`--tail` print to stdout for mobile touch-scroll ŌĆö see the mobile-read note below), `on` (native; two-layer quoting + `zsh -lic` contract), and delegates `plan`/`paste`/`kill` to the `_t_*` zsh helpers. Internal `session-rows`/`land`/`kill-owner` expose the remote contract.
  - **`t()` in `.zshrc`** is the front door. Bin-native verbs run straight through via **`command t`** (the `command` builtin reaches `~/bin/t` past the function ŌĆö the same name-collision trick `claude()` uses). The **shell-bound** verbs (`open`/`pop`/`push`/`find`) map to the existing zsh helpers, which already do the `cd` + `claude -r` in the *current terminal* and the tpush sentinel handoff correctly **because they run in the calling shell** ŌĆö a bin subprocess can't. So there is **no resolve protocol**; the shim just calls them, and the one-live-owner invariant is literally still `_t_push`/`claude()` unchanged.
  - **Hard cutover done:** the 9 old top-level names are **gone** (typing `dev`/`tpush`/ŌĆ” ŌåÆ command not found). Their implementations were renamed to internal helpers ŌĆö **`devŌåÆ_t_dev`, `tpushŌåÆ_t_push`, `tpopŌåÆ_t_pop`, `tbeamŌåÆ_t_beam`, `tplanŌåÆ_t_plan`, `tpasteŌåÆ_t_paste`, `tfindŌåÆ_t_find`** ŌĆö and `tread`/`on` were reimplemented natively in `bin/t`. **The detailed notes below still describe this logic and its invariants accurately; only the function names + how they are invoked changed.** Grammar mapping: `devŌåÆt open`, `dev <repo> newŌåÆt open <repo> --new`, `dev ŌĆ” -fŌåÆt open ŌĆ” --fg`, `dev lsŌåÆt ls`, `dev killŌåÆt kill`, `dev -rŌåÆt open --remote` (now also auto-detected ŌĆö a slot live only on another host is attached in place without the flag), `dev -r ŌĆ” --hereŌåÆt beam <repo> <slot> --from <host>` (the pull/receive moved to tbeam), `dev -r killŌåÆt kill --remote`, `dev <repo> fgŌåÆt open <repo> fg` (the separate `t fg` verb was folded into `t open`: `t open <id>` / `t open <repo> <id>` / `t open <repo> fg` all adopt a foreground session here ŌĆö an id-shaped arg, 4+ hex chars or the displayed `repo:id` label, dispatches via `_dev_fg_handle`; pure digits stay slot numbers), `tpushŌåÆt push`, `tpopŌåÆt pop`, `tbeam [host]ŌåÆt beam --host <h>`, `treadŌåÆt read`, `tplanŌåÆt plan`, `tpasteŌåÆt paste`, `tfindŌåÆt find`, `onŌåÆt on`.
  - **Repo-aware positionals (every slot verb):** any verb taking `<repo> [slot]` infers the repo from `$PWD` when called bare or with a lone numeric slot ŌĆö `t paste 4` in `~/code/financial-forecast` pastes into *that* repo's slot 4; bare `t open` opens/reattaches the repo you're standing in; likewise `t open 4`/`--new`, `t kill 4`/`t kill all`/bare `t kill` (infers the repo, still demands a slot), `t pop 4` (and bare `t pop` now matches from a subdir ŌĆö repo dir, not exact `$PWD`), `t plan 4`, `t read 4`/bare `t read`, `t beam 4 [--host h]`/`t beam [4] --from <host>`, and no-arg fg adoption (`t open fg`) prefers this repo's foreground claudes. Inference is **`_t_infer_repo [slot]`** (.zshrc, next to `_dev_cwd_repo_dir`): a **live `dev-*` session rooted in the cwd's repo dir wins** ŌĆö exact slot's session first, then any live one ŌĆö with the alias read off the actual session *name*, so it matches however the slot was created when several aliases key one dir (`dev-dot-4` vs `dev-dotfiles-4`); no live session ŌåÆ the dir's DEV_REPOS key (basename match, then shortest). `t read` is bin-native, so `bin/t` has the twin **`_infer_repo`** working over the log files instead (exact-slot log, then any `dev-<alias>-N.log`, **newest mtime wins** ŌĆö logs outlive sessions, so stale-alias logs lose to the alias in use). Deliberately **not** inferred: the bare `t open -r` every-remote-slot picker (documented behavior), and `t beam --from` uses alias-only inference (the slot lives on the *remote* host, so a same-numbered local session name is no evidence).
  - **Config bridge:** `bin/t` can't read zsh assoc arrays, so `_t_sync_config` (in `.zshrc`) derives **`~/.config/t/config.sh`** (plain `NAME[key]=value` lines) from the live `DEV_REPOS`/`DEV_BRANCHES`/`REMOTE_HOSTS`/`DEV_WORKTREE`/`DEV_BRANCH`/`DEV_WORKTREE_ROOT`/`DEV_WORKTREE_DEFAULT`, mtime-gated; `bin/t` parses it with a strict regex (never sources it; `~/.zshrc.local` stays the source of truth and is never executed by the bin). zsh keeps using the live arrays for the cd/host shortcuts + the `_t` completion.
  - **Remote contract unchanged:** the cross-host fan-out still runs `_dev_session_rows`/`_dev_rows_all`/`_tbeam_land`/`_tbeam_kill_owner` over `ssh ŌĆ” zsh -lic` ŌĆö those are `_`-prefixed internals that **survive the cutover** (the `zsh -lic` is required because `~/bin` + Homebrew are only on the interactive PATH). Remote `_dev_remote` attach/kill now run `t open`/`t kill` on the far host (which has only `t`). **Adding `bin/t` to a host needs `install.sh` there, not just `dots`** (`dots` pulls + reloads but does not relink a *new* bin script); the rc-127 "stale dotfiles" note in the fan-out is the in-band reminder. **Minor known gap:** `t on` does not set the Terminal tab title (the old `on` called `_term_title`) ŌĆö a cosmetic follow-up.
- **`bin/csync`** ŌĆö `set -euo pipefail` Bash that two-way-merges Claude session history (`~/.claude/projects` Ōåö iCloud Drive `claude-sessions`) **and saved plans** (`~/.claude/plans` Ōåö `claude-plans`) through a shared `sync_pair` helper (`rsync -a --update`, no `--delete`, so machines converge to the union of both trees). Plans are synced so `t plan` finds a plan on any machine ŌĆö not just the host that authored it; before this, a session beamed to another host left its plan `.md` orphaned on the origin (the transcript moved, the plan file did not), and `t plan` reported "no saved plan" (see the `t plan` note). Was a `.zshrc` function; extracted to a script (`help` still lists it ŌĆö it scans `~/bin`). `brctl download` is wrapped in `|| true` so its exit status never aborts the sync under `set -e`. **Periodic runs come from a `precmd` hook in `.zshrc` (`_csync_periodic`), not launchd** ŌĆö this is load-bearing and a dead end was hit getting there: iCloud Drive is TCC-protected and *background launchd agents are denied*, and on recent macOS (Tahoe) granting `/bin/bash` Full Disk Access has **no effect** (verified ŌĆö even a pure-bash `readdir` is denied under a launchd agent), because TCC won't pin an FDA grant to a platform interpreter running an arbitrary script. The shell hook sidesteps this entirely by running `csync` in the Terminal's already-approved context. The hook gates on a `~/.cache/csync-last-run` stamp (written *before* the run so overlapping shells don't double-fire), uses `$EPOCHSECONDS` (no `date` fork per prompt), and launches `csync` detached (`( csync ŌĆ” & )`) so it never blocks the prompt; background runs log to `~/Library/Logs/csync.log`. Don't reintroduce a launchd agent for this without a code-signed helper binary.
- **`bin/pr-watch`** ŌĆö Python 3 (gh-style argparse, like `bin/t`); an **autonomous PR fixer** that runs as a **launchd LaunchAgent** (`launchd/com.chrisobrien-ai.pr-watch.plist`, `StartInterval` 300). It polls `chrisobrien-ai/dotfiles` (override `$PR_WATCH_REPO`) via `gh pr list --json`, and for any open, non-draft, same-repo PR that is `CONFLICTING` **or** has a concluded CI **FAILURE**, it spawns a detached tmux Claude session (`pr-dotfiles-<N>`) that resolves it and pushes ŌĆö then **notifies** (macOS `osascript` notification + a `gh pr comment` carrying the `tmux attach` command) so you can intervene. Verbs: `poll [--dry-run]` (the agent entrypoint), `enable`/`disable` (the opt-in flag + `launchctl bootstrap`/`bootout`), `status`, `clean`. **Why launchd is right here when it was a dead end for csync** (see the `bin/csync` note above): csync abandoned its launchd agent because iCloud Drive is TCC-protected and background agents are denied ŌĆö `pr-watch` touches **no** TCC-protected dir (only gh/git/tmux), so the agent timer is the correct tool (a LaunchAgent, not Daemon, because it needs the GUI session for tmux + notifications). **Autonomy guardrails (it edits code and pushes on its own):** (1) **opt-in** ŌĆö `poll()` no-ops unless `~/.config/pr-watch/enabled` exists, so `install.sh` materializes the plist **inert** and a fresh clone never auto-arms a pusher (install.sh only `bootstrap`s the agent on a machine already opted in, so re-running it picks up plist changes without arming a new box); (2) **once per `(PR number, head SHA)`** ŌĆö recorded in `~/.cache/pr-watch/state.json`, re-eligible only when new commits land; (3) **attempt cap** (`MAX_ATTEMPTS=2` per PR) ŌåÆ then it gives up + notifies for manual attention (no fixŌåÆfailŌåÆfix loop); (4) **live-session guard** ŌĆö never a second session for a PR already being fixed; (5) **isolated worktrees** ŌĆö each fix runs in `~/.cache/pr-watch/worktrees/<repo>-pr<N>` at a **detached HEAD**, so the PR branch may be checked out elsewhere (e.g. `dev/claude-1` in the live tree or a concurrent agent's checkout) without a collision, and it pushes via `git push origin HEAD:<branch>`; the live symlinked tree is never touched. The task brief is written **outside** the worktree (`~/.cache/pr-watch/tasks/`) so the session can't accidentally commit it, and the session runs `claude --dangerously-skip-permissions` (the point ŌĆö no approval prompts) while staying attachable for oversight. Logs to `~/Library/Logs/pr-watch.log`. **Kill switch:** `pr-watch disable` (removes the flag + boots out the agent). Note: this repo has no test suite ŌĆö "failing tests" means failing **CI checks** (shellcheck/gitleaks/pii-scan/Bugbot); `pr-watch` reacts to any concluded `statusCheckRollup` FAILURE generically.
- **`bin/sleep-manager`** ŌĆö `set -euo pipefail` Bash with a `case` dispatcher. Sleep blocking is two layers: `pmset -a disablesleep 1` (OS-level, survives logout/reboot) plus a backgrounded `caffeinate -dimsu` whose PID is tracked in `/tmp/sleep-manager-caffeinate.pid`. Key gotcha documented in its `--help`: the caffeinate process dies on reboot but the pmset flag does **not** ŌĆö always run `enable` to fully restore; don't rely on a reboot. `status` is the source of truth for current state.
- **`.zshrc`** ŌĆö PATH (`~/bin` first), then `compinit` is run **before** sourcing anything that calls `compdef` (e.g. a completion sourced from `~/.zshrc.local`) ŌĆö order matters, this was a fix for a "command not found: compdef" error. Also defines `prview` (gh PR check summary via jq) and the interactive `nosleep` one-liner (the inline cousin of `sleep-manager`). **Per-machine config is split out:** `.zshrc` declares an empty `DEV_REPOS` and sources `~/.zshrc.local` (a real copy, not symlinked ŌĆö see the symlink-model note), which holds the real repo paths, `export TBEAM_HOST`, and any private completions. The committed `.zshrc` ships generic examples and derives all usage/error/completion strings from `${(k)DEV_REPOS}`, so it works (and stays generic) even with no local file. `install.sh` seeds `~/.zshrc.local` from `.zshrc.local.example`. **Help is single-source-of-truth, split into extractor + renderer:** the doc-comment block directly above a `name()` function *is* that command's `-h`/`--help` text. **Exception ŌĆö the `t` verbs do NOT use `_help_for`:** the `t()` shim routes *any* `-h`/`--help` to `command t`, so every `t <verb> -h` renders from **bin/t's argparse** (each subparser carries a `description=`), and the `_t_*` zsh helpers carry **no** `_help_for` dispatch ŌĆö the old `tpush [-p]`-style help blocks above them were pruned to plain comments (re-adding a `_help_for _t_X` would be dead code the shim never reaches). `_help_for` now serves only the genuinely shell-native commands ŌĆö `prview`/`nosleep`/`dots`/`help`. (The dynamic no-arg/bad-arg paths below still call **`_help_style`** directly ŌĆö they fire on a bare/bad `t open` etc., not on `-h`, so the shim does not intercept them.) **`_help_for name`** `awk`-extracts the contiguous `#` block immediately above `name() {` (a **truly blank** line resets it; a bare `#` is kept as a spacer, so rationale can sit above the help separated by one empty line), strips the `# `, and pipes it through **`_help_style`** ŌĆö the standalone gh-style RENDERER (split out of `_help_for` so the dynamic paths below can reuse it). `_help_style` reads plain text on stdin and styles a gh-style convention: **line 1** `name ŌĆö tagline` (name bolded on the ` ŌĆö `), a `Usage:` line (label bold; an indented *second* Usage line stays verbatim under it), bare `Word:` section headers (`Commands:`/`Options:`/`Arguments:`), and **2-space-indented `term  <2+sp>desc` rows** (term cyan, desc dim, descriptions **auto-aligned** to the widest term so comment spacing needn't be hand-tuned). A **wrapped description line** ŌĆö indented **3+** spaces right after a kv row (or its own continuation), tracked by an `eligible` flag so a Usage/paragraph follow-on is excluded ŌĆö is hang-indented to the same description column (this is why a row's continuation comment no longer needs hand-alignment; just indent it). Color is gated on `[[ -t 1 ]]` in `_help_style`, which is the **last** stage of every help pipe, so the test sees the real terminal and `cmd -h | cat` stays plain. To add/restyle a command's help, just write its comment in that shape ŌĆö no per-command formatting code. **The dynamic no-arg/bad-arg paths (`dev`, `dev kill`, `tread`) now build their text ŌĆö including the *actual* `${(k)DEV_REPOS}` / live logs that static help can't show ŌĆö and pipe it through `_help_style` too, so they render identically to `-h`** (no more ad-hoc `echo "Usage: ŌĆ”"`). **Gotcha:** `_help_style`'s awk body is a single-quoted string, so **no apostrophes in its inline comments** (`is NOT`, not `isn't`) ŌĆö one closes the quote and breaks the source.
- **`tfind` (in `.zshrc`)** ŌĆö semantic search across saved Claude sessions: "which session was working on X?". Pipeline is **retrieve-then-rerank** ŌĆö a keyword pass over each transcript's title + your prompts builds a candidate pool (the same scan `_claude_sessions_fzf` uses; padded with recent sessions when keyword matches are thin, so divergent wording still reaches the model), then **Sonnet via `claude -p --model sonnet --output-format json`** reranks the candidates and returns a JSON `[{i, why}]` ranking. Two load-bearing details: (1) the model call is a **Python `subprocess.run(['claude', ŌĆ”])`**, which execs the real `~/.local/bin/claude` binary and so **bypasses the zsh `claude` wrapper** (the wrapper's tpush sentinel would otherwise interfere); (2) **every failure path ŌĆö no CLI, timeout, unparseable reply, empty `[]` ŌĆö falls back to keyword order** so the fzf picker always populates and the command works offline. `tfind -k` forces the offline keyword path (no Sonnet). The keyword scorer (`hay.count(t) + 4*head.count(t)`) is recall-only here ŌĆö Sonnet does the precision ŌĆö so don't over-tune it.
- **`tbeam` (in `.zshrc`)** ŌĆö **moves** a Claude session between machines; a **cross-host `tpush`/`tpop`**. **Both directions live here:** by default it **sends** the resolved session to a host (default `$TBEAM_HOST`, set in `~/.zshrc.local`) and resumes it there; **`tbeam <repo> [slot] --from <host>`** **receives** ŌĆö pulls a session that lives on `<host>` down into a local dev slot (the `--from` branch is parsed early in `_t_beam` and delegates to **`_dev_pull`**, which was the engine behind the old `dev -r ŌĆ” --here`; repo is required, slot optional ŌåÆ the host's first live slot). This is the home of all session *moving*; merely *attaching* a remote slot in place (no move) is `t open`'s auto-detect, not tbeam. The SEND path resolves a session exactly like `tpush` (current chat via `$CLAUDE_CODE_SESSION_ID`, else fzf-pick scoped to `$PWD`, `-a` for all; **`-s`/`--session <id>`** targets an explicit id/unique-prefix, resolved against local transcripts by globbing `~/.claude/projects/*/<id>*.jsonl` for a unique match then `_tbeam_transcript_cwd` for its recorded cwd), `rsync`s its `~/.claude/projects/<enc>/` dir over (`_tbeam_sync_transcript`, `--update`/no-`--delete` so the live source wins but a newer remote is never clobbered ŌĆö csync still reconciles in the background, transcripts and now plan files alike, so a plan authored before the move catches up), then has the **host land it**. Landing logic (`_tbeam_land`) lives in the *shared* dotfiles so it runs on the far side via `ssh "$host" "TB_CWD=ŌĆ” TB_SID=ŌĆ” TB_MODE=ŌĆ” zsh -lic _tbeam_land"` ŌĆö work passed in **`TB_*` env vars** (sidesteps nested-quote hell) and a **login+interactive** shell (`zsh -lic`) because Homebrew `tmux`/`fzf` are on the *login* PATH (`.zprofile`) while `claude` is on the *interactive* PATH ŌĆö plain `ssh host 'ŌĆ”'` sees neither. Default mode reuses the host's own `_dev_resume_session` to make a first-class `dev-<repo>-<slot>` (so `dev`/`tread`/`tpop` work there), then auto-attaches via `ssh -t`; `-f` resumes in the ssh foreground, `-d` leaves it detached with an attach hint (`t open <repo> <slot>` to attach it in place, or `t beam <repo> <slot> --from <host>` to pull it back). Two load-bearing constraints: (1) the repo must exist at the **same path** on the host (all repos live in `~/code` on every machine) ŌĆö `tbeam` guards with `ssh host test -d`; (2) from *inside* Claude it forces `-d` (a Bash subprocess has no TTY to `ssh -t` into). `_tbeam_land` is shared code, so after adding/changing it run `dots` on the host to pull it. **tbeam is a move, not a copy:** once the session lands on the far side, the origin's live owner is *stopped* so exactly one live claude owns the id (two live owners of one transcript have no lock and diverge). The SIGTERM-self step gates on **`self_move`** (`sid == $CLAUDE_CODE_SESSION_ID`), not merely "inside Claude" ŌĆö so `tbeam -s <other-id>` run from inside a Claude session stops *that id's* dev slot via the shared `_tbeam_kill_owner` and never kills the foreground claude you're typing in; a true current-session move SIGTERMs the foreground claude after landing (mirrors `tpush`'s auto-exit). `_tbeam_kill_owner` is shared too (also used by the `--from` receive in `_dev_pull`), another reason the move needs `dots` on the host. (There is no `--no-tmux` flag ŌĆö `-f` is the no-tmux mode: foreground resume, no dev slot.) To *survey* what's live across machines before moving, see **`t ls -r`**; **`t open <repo> <slot>`** attaches one in place (auto-finding its host) and **`t beam <repo> <slot> --from <host>`** pulls it down here.
- **`t open` remote attach ŌĆö auto-detect + explicit `-r` (in `.zshrc`, `_t_dev`)** ŌĆö `t open` is the one "get into a session" verb, local or remote, and **it never moves a session ŌĆö it attaches in place wherever the slot lives** (moving is `tbeam`'s job). **The repo arg is optional** ŌĆö bare `t open` in a `DEV_REPOS` dir defaults to that repo via `_t_infer_repo` (see the repo-aware positionals note above; the merge dropped mainŌĆÖs simpler `_dev_cwd_repo_key` twin); a lone slot (`t open 2`/`t open fg`/`t open new`) with no valid repo likewise targets the cwd repo. **Ordering matters:** the auto-detect probe runs *after* this repo-resolution (and after fg-adoption returns), so even a bare `t open` resolves its repo first and can then find a slot that is live only on another host ŌĆö the probe sat before repo-defaulting in the original commit and was moved down when it merged with the cwd-default work. Resolution order in `_t_dev`: a **live LOCAL slot always wins** (cheap `_dev_local_slot_live` tmux check, no ssh); only when the named slot is **not live here** does it probe `$REMOTE_HOSTS` (host auto-inferred via `_dev_remote_resolve`/`_dev_rows_all`) ŌĆö live there ŌåÆ **attach in place** over `ssh -t <target> "zsh -lic 't open <repo> <slot>'"` (the session stays put, the old `tgo`); live nowhere ŌåÆ fall through to the **local fresh-start** path. The **auto-probe is gated** to keep the common path cheap: skipped when no `$REMOTE_HOSTS` are configured (free on a single-host box), for an explicit fresh/foreground request (`new`/`fg`/`-f`), and outside `DEV_REPOS` ŌĆö and it only runs at all when nothing is live locally for that repo/slot. **`-r`/`--remote` is the EXPLICIT branch** (`_dev_remote`): force the remote action even when a local slot exists, and the only way to reach the **bare `t open -r` ŌåÆ every remote slot** picker (`t open -r api 5` ŌåÆ that one, `t open -r api` ŌåÆ every `api` slot). One candidate ŌåÆ use it; several ŌåÆ **fzf-pick** (`host/slot  summary`, needs TTY+fzf); none ŌåÆ bail with a `t ls -r` hint (now also hinting `-r --new` to *start* one). **`-r --new` STARTS a fresh remote session** (the attach path is for live slots; `--new` flips it to create): with no `--host` it picks the **default host = `_dev_default_host`, the alphabetically-first `$REMOTE_HOSTS` key** (deterministic since the assoc array is unordered), printing which host it chose; `--host <h>` overrides. This is dispatched in `_t_open` *before* `_t_dev` (remote + `--new` + no `--host` ŌåÆ set host to the default, then fall into the same `_dev_remote_open` path as `--host`), so `-r` and `--host` are **two angles on one model** ŌĆö `--host` names the host, `-r` means "remote, you pick" ŌĆö and they compose (`-r --host h ŌēĪ --host h`; the redundant `-r` is dropped before forwarding so the far side acts locally). The chosen slot name (`<repo>-<num>`, split on the *last* dash so multi-dash repos like `dotfiles` resolve) gives repo+slot. A no-TTY guard prints the command instead of failing. **Starting a NEW session remotely: `t open <repo> [slot] -r --new`** (default host) **or `--host <h>`** (a specific one) ŌĆö both reach `_dev_remote_open`, dispatched from the `_t_open` shim before `_t_dev` ever runs locally. Unlike the `-r` *attach* / auto-detect paths ŌĆö which only reach an *already-live* slot ŌĆö these force a host and so can spawn a fresh remote session (`t open ff -r --new`, `t open ff --host mini`). It forwards the original gh-style args (minus `--host`) over `ssh -t <target> "zsh -lic 't open ŌĆ”'"`, so the far host's own `t open` re-parses them; no-TTY guard, session stays on `<host>`. **Pulling a remote session HERE is no longer an `open` flag ŌĆö it is `t beam <repo> <slot> --from <host>`** (the receive direction, `_dev_pull`; see the tbeam note). **`t kill -r <repo> [slot]`** tears a slot down on its host ŌĆö resolve ŌåÆ `ssh -t <target> "zsh -lic 't kill <repo> <slot> [-y]'"` (remote confirm works through the TTY unless `-y`). `_dev_remote_resolve` (prints `host\trepo\tslot`) is shared by the explicit `-r` attach, the auto-detect attach, and `kill`; `_dev_remote_attach` (resolve once, then ssh -t attach) is shared by the explicit and auto paths so the slot is scanned only once. Because hosts run `_dev_session_rows`/`_tbeam_kill_owner` over ssh, remote `t open` needs their dotfiles current ŌĆö run `dots` there (same caveat as `t ls -r`). **Remote-aware read verbs:** `t plan` (and any future per-slot reader) is host-bound because its artifact is host-local ŌĆö the plan `.md` lives in the slot's `~/.claude/plans`. csync now converges plan files across machines too (it no longer syncs only transcripts), but only periodically (~15 min), so a freshly-beamed slot's plan may not have propagated yet ŌĆö `t plan ff 1` on a slot beamed to mini still renders *on mini* so it shows the authoritative current plan without waiting on a sync. `_t_plan`'s "No such session" path now calls the shared **`_dev_remote_delegate <repo> <slot> <verb> [extraŌĆ”]`** (next to `_dev_remote_attach`): if the slot is not live locally but resolves to a remote host, it runs `t <verb> <repo> <slot>` there over `ssh -t` and returns 0; else returns 1 and the caller falls back to its local path. Same gating as the auto-attach (no `$REMOTE_HOSTS` ŌåÆ no-op; no-TTY ŌåÆ print a hint and still return 0 so it does not fall through to a local error). **Every per-slot verb routes through one shared check** ŌĆö **`_dev_session_remote_fallback <session-name> <verb> [extraŌĆ”]`** (next to `_dev_remote_delegate`): it splits repo/slot off the `dev-<repo>-<slot>` name (last dash, so multi-dash repos resolve) and hands them to `_dev_remote_delegate`, returning 0 = handled on the host / 1 = no remote slot. `_t_pop`, `_t_plan`, and `_dev_kill` all call it on their local-miss path, so the parse is not re-open-coded per verb and detection is uniform. **What each verb does on a remote hit:** `t pop` delegates `pop` (un-tmuxes *on the remote*; you drive it through the `ssh -t` TTY ŌĆö same semantics as a local pop) ŌĆö and **bare `t pop`** (no args), after finding nothing local, infers the repo and probes with an *empty* slot (`_dev_remote_delegate <repo> "" pop`, which fzf-picks among the repo's remote slots), so it is no longer local-only. **`t kill <repo> <slot>`** (no `-r`) now auto-detects too: not live here but live remotely ŌåÆ delegate `kill` to the host (the remote `t kill` still confirms unless `-y`, which propagates via `${force:+-y}`); a repo with *no* local sessions and no named slot just prints a `t ls -r` / `t kill -r` pointer rather than guessing a kill target. `-r` remains the explicit "force remote / bare picker" form. **Killing a FOREGROUND / non-dev-slot session (the `<repo>:<id>` / `:fg` rows `t ls` shows via `_dev_fg_rows` ŌĆö a bare `claude`, or a pr-watch `pr-dotfiles-N` session):** `t kill <id>` / `t kill <repo>:fg` locally, `t kill -r <id>` remotely. These are NOT dev slots, so the whole dev-slot machinery (`_dev_kill`'s name grep, `_dev_remote_resolve`, which both drop `:`-labelled rows) skips them ŌĆö the old gap where `t kill dotfiles-pr47` said "no session." The companion path is **`_dev_kill_fg`** (local) + **`_dev_remote_fg_kill`** (cross-host, via `_dev_rows_all`): it re-derives each fg row's label the same way `_dev_fg_rows` does (scanning live `claude` pids, excluding dev-slot claudes and this shell), matches `<id>` against the exact label, the label's repo part *only when that part is not itself a `DEV_REPOS` key* (so `t kill dotfiles-pr47` hits the fg row but `t kill dotfiles` still lists dev slots), or a bare session-id prefix, then kills via **`_dev_tmux_session_of_pid`** ŌĆö a claude inside a tmux session (pr-watch) ŌåÆ `tmux kill-session`; a true no-tmux foreground claude ŌåÆ SIGTERM (clean, transcript stays resumable, same as tpush). Both return **rc 2 on no-match** so the caller only falls through to the dev-slot resolver on a true miss, not after a declined confirm. Wired into `_dev_kill` / `_dev_remote_kill`'s no-local-slot branch; `t ls`'s footer advertises `kill: t kill <id>`. **`t read`** is bin-native, so `cmd_read` (on a missing local log) calls a **non-exec** sibling **`_try_remote_delegate`** in `bin/t` ŌĆö it runs `_dev_remote_delegate` as a *child* (inherits the TTY, so the remote pager works) and **returns** the rc instead of `os.execvp`-ing, so it can fall back to the local "No log found"; **bare `t read`** (no slot) probes with an *empty* slot (`named_slot or ""`) so a remote slot other than the local default `1` is still found. All of this leans on `_dev_remote_delegate` returning 0 after the `ssh` regardless of the remote verb's exit. **Mobile reading ŌĆö `t read -d`/`--dump` and `-n N`/`--tail` (in `bin/t`, `cmd_read`):** `less` is unusable on a phone terminal (Termius etc.) ŌĆö it repaints one viewport and needs arrow/PageUp keys, and you cannot finger-scroll it; only text printed **straight to stdout** lands in the terminal's *native* touch-scrollable scrollback. `-d`/`--dump` prints the log to stdout (no pager); `-n N`/`--tail` (bare `-n` = last 200) keeps only the last N lines and **implies `--dump`**. **Load-bearing detail ŌĆö both the `less` path AND the dump path share one cleaner, `_clean_capture` (in `bin/t`); the old `_PERL_STRIP`/`perl` pipe is gone** (so is the `perl` dependency). It was replaced because the perl stripper garbled real TUI logs two ways: (1) it deleted `\r`, but a tmux `capture-pane` log of a TUI separates screen REPAINTS with bare `\r` (overwrite column 0) and emits a real `\n` only rarely ŌĆö so stripping `\r` first collapsed the whole repaint history into a few giant `\n`-lines (`--tail 3` once grabbed a 1.4MB blob, and `less` showed words jammed together); and (2) its hand-picked CSI-letter allowlist missed the private-mode sequences a modern TUI emits ŌĆö `\x1b[>4;2m` / `\x1b[>1u` / `\x1b[<u` (keyboard protocol), `\x1b[?25h/l` (cursor) ŌĆö whose `>`/`<`/`?` parameter marker defeated the `[\d;]*` match, so they leaked into `less` as `^Qu1u4;2m`-style junk (plus `^N`/`^O` SO/SI charset-shift bytes showing as literal control glyphs). `_clean_capture` parses the **full CSI grammar** (`ESC [` <params `0x30-0x3F`> <intermediates `0x20-0x2F`> <final `0x40-0x7E`>), dropping every CSI **except plain SGR colour** (final `m`, digit/semicolon params only) so `less -R` still renders colour; strips OSC title-sets, charset selects, and leftover C0 control chars (keeping `\t`/`\n`/`\r`/`\x1b`); and **splits on BOTH `\r` and `\n`** so each repaint row is one display line and the tail is bounded + readable. **Crucially it does all stripping on raw BYTES before the utf-8 decode** ŌĆö a capture interleaves SO/SI bytes and escapes *inside* multibyte chars (`ŌÅĄ`, `ŌöĆ`, `ŌåÉ`), so decoding first and stripping after left the char already mangled into `’┐Į`; stripping the control/escape bytes first makes each char's bytes contiguous so it decodes cleanly. (Residual word-jamming like `monthpriceforecast` is the TUI positioning words with cursor moves we strip; reconstructing it needs a terminal emulator and is out of scope.) Both flags **forward to a remote slot** via `_try_remote_delegate`'s new `extra` arg (ŌåÆ `_dev_remote_delegate ŌĆ” [extraŌĆ”]`), so `t read ff 3 -n` renders the mobile dump on the slot's host. **Deliberately NOT remote-aware: `t paste`** ŌĆö it sends a *local* iCloud file into a session, so delegating would paste from the *remote's* iCloud (shipping the file too is a separate, heavier feature); it stays local-only. **`t push`** is inherently local (it pushes the foreground session you are typing in).
- **`dev ls -r` / cross-host `dev ls` (in `.zshrc`)** ŌĆö `dev list`/`dev ls` with `-r`/`--remote` renders `_dev_list` across **this machine AND every `$REMOTE_HOSTS` host** in one table (same `ŌŚÅ attached ┬Ę Ō£ō active context` header) with a dedicated **HOST column** (STATUS/HOST/SESSION/WORKING ON) ŌĆö local rows read `local`, remote rows their `$REMOTE_HOSTS` key. It's read-only ŌĆö a survey of "what's running on macbook vs mini"; you *attach* a remote one in place with `t open <repo> <slot>` (host auto-inferred ŌĆö no flag needed) or *pull* it down with `t beam <repo> <slot> --from <host>` (which enforces the move + one-live-owner invariant). **The key design call is *why it lists live dev slots, not transcripts*:** transcripts already converge on every machine via csync, so an all-transcripts cross-host view would be mostly duplicate rows (the same id once per host) ŌĆö whereas live `dev-<repo>-<slot>` slots genuinely differ per machine. Three helpers: **`_dev_session_rows`** is the machine-readable sibling of `_dev_list` (one `sid\tcwd\tslot\tstate\tcontext\tsummary` row per live slot, reusing `_dev_session_has_claude`/`_dev_session_at_welcome`/`_dev_session_summary`; an unstamped/idle slot's empty `sid` is sentinel'd `-` because **tab is an IFS-whitespace char**, so a leading/consecutive empty field would otherwise collapse under `read` and slide the columns); **`_dev_rows_all`** fans it out over `local` + every `$REMOTE_HOSTS` value **in parallel** (background ssh ŌåÆ temp files ŌåÆ `wait`) with `ConnectTimeout=3`/`BatchMode=yes`, stashing each host's ssh exit code in a `.rc` file to distinguish **unreachable** (rc 255 ŌåÆ "unreachable") from **reachable-but-stale-dotfiles** (rc 127 ŌåÆ "run `dots` there"), both skipped + noted on stderr so one sleeping Mac never stalls the table ŌĆö then prefixes each row with its host **verbatim** (no re-split, so empty fields still can't collapse); **`_dev_list_remote`** re-renders those rows in `_dev_list`'s format under a HOST column (`$COLUMNS`-truncating WORKING ON). Because each host runs `_dev_session_rows` over ssh, `dev ls -r` needs every host's dotfiles current ŌĆö run `dots` there (same caveat as `dev -r`); the rc-127 "stale dotfiles" note is the in-band reminder. **Scoped by default:** both `dev ls` and `dev ls -r` show only the repo you're standing in ŌĆö `_dev_cwd_repo_dir` maps `$PWD` to its `DEV_REPOS` dir (path-based, so `dot-*` and `dotfiles-*` both show in `~/code/dotfiles`) and the renderers filter rows by that dir; `-a`/`--all` widens to every repo, and standing outside any repo is unscoped. **Foreground sessions too:** `_dev_fg_rows` adds claudes you ran directly in a terminal (live `comm == claude` not in a `dev-*` tmux pane, minus the one *this* shell runs under), labeled `<repo>:fg`, with sid+cwd from the claude-stamp-tmux registry (see that note) so the title is exact; it's appended to `_dev_session_rows` (so `dev ls -r` covers each host's foreground claudes) and rendered by `_dev_list`.
- **`on` + `REMOTE_HOSTS` (in `.zshrc`)** ŌĆö `on <host> [cmdŌĆ”]` runs a command on another machine over `ssh -t "$host" "zsh -lic <cmd>"`. The **`zsh -lic`** (login+interactive) is the same trick `_tbeam_land` relies on: Homebrew `tmux`/`fzf` are on the *login* PATH while `claude` and the dotfiles functions are on the *interactive* PATH, so a plain `ssh host 'ŌĆ”'` sees neither ŌĆö `-lic` makes `dev`/`tread`/etc. resolve remotely (e.g. `on mini dev dot` starts a dev session there). No command ŌćÆ an interactive shell on the host. The **command quoting is two layers** (`local cmd="${(j: :)${(@q)@}}"` then `ssh ŌĆ” "zsh -lic ${(q)cmd}"`): `(@q)` quotes each arg so the remote `zsh -c` sees the original words, then `(q)` wraps the joined string as **one** token so ssh ŌĆö which re-splits its remote command on spaces ŌĆö transports it intact (verified: a `;` in an arg arrives as data, not a separator). Hosts live in **`REMOTE_HOSTS`** (assoc array, declared by `.zshrc`, populated in `~/.zshrc.local` ŌĆö same single-source-of-truth pattern as `DEV_REPOS`): each key auto-generates a **shorthand function** (`mini ŌĆ”` ŌēĪ `on mini ŌĆ”`, made via `functions[$key]=ŌĆ”` in a loop next to the `DEV_REPOS` cd-shortcut loop) plus `_on_hosts` tab-completion and a synthesized `help` entry (the generated functions aren't real text in the file, so ŌĆö like the cd shortcuts ŌĆö `help` synthesizes their entries from the array). An **unknown host is used as a literal ssh target** (`${REMOTE_HOSTS[$h]:-$h}`), so `on box.local uptime` works unregistered. **Back-compat seed:** an old `MINI_HOST`/`TBEAM_HOST` populates `REMOTE_HOSTS[mini]` when the registry has no `mini`, so a `mini` shortcut keeps working with no config change. (`on` is defined far below the generation loop, but function bodies bind late, so a generated `mini` calling `on` is fine.)
- **`tpush`/`tpop`/`claude()` wrapper (in `.zshrc`)** ŌĆö move a Claude session between a foreground terminal and a detached `dev-<repo>-<slot>` tmux session. **The load-bearing invariant: never two live `claude` processes on one session id.** Claude Code takes *no lock* on a session's `.jsonl` and does *not* auto-fork ŌĆö two live resumers of the same id both append with no coordination and the conversation diverges (the copy you aren't driving freezes). So: (1) `tpush` from *inside* Claude does **not** spawn `claude -r` itself ŌĆö it writes a `SPAWN\t<session>\t<cwd>\t<sid>` instruction to the `$CLAUDE_TPUSH_ATTACH` sentinel, and the `claude()` wrapper spawns the detached resume **after** the foreground exits (when the id is free) and then attaches. **The exit is now automatic:** after writing the sentinel, `tpush` walks the process ancestry (`_tpush_claude_pid`) to the nearest `claude` and `kill -TERM`s it, so you no longer type `/exit` ŌĆö SIGTERM exits Claude cleanly in ~2s (transcript is appended live, so only the in-flight tpush turn is lost; terminal is restored on the wrapper's `tmux attach`). The kill fires *after* the sentinel is written and *before* any spawn, so the one-live-owner invariant holds; if the ancestor walk fails it falls back to printing the manual `/exit` hint. **Don't reintroduce a pre-spawn here** ŌĆö the kill-then-wrapper-spawn ordering is what keeps a single owner. (`ATTACH\t<session>` is the already-backgrounded case; a plain-shell picker spawns inline since the picked session isn't live; an older shell with no wrapper spawns eagerly + warns.) (2) `tpop` captures the tmux session's claude PID (pane shell's child), `kill-session`s, then **waits for that PID to actually exit** (Ōēż5s) before `claude -r` in the foreground ŌĆö `kill-session` only SIGHUPs, so resuming immediately raced the dying process. Was the "popped/backgrounded session stops updating while the original keeps going" bug. **`dev <repo> <slot> -f` delegates here:** in `dev`, `-f` (no-tmux) foreground-RESUMES the named slot if it's live (ŌēĪ `tpop <repo> <slot>`), and only starts a *fresh* inline claude when that slot doesn't exist ŌĆö so `-f` means "foreground" consistently across `dev`/`tbeam`/`dev -r`, not "fresh" in one and "resume" in the others.
- **`_dev_worktree_*` ŌĆö worktree-per-session (in `.zshrc`); THE structural fix for sibling collisions.** Every NEW dev session gets its own git worktree on its own branch instead of all `dev-<repo>-*` slots sharing one tree on `dev/claude-1`. This is the "deeper fix = per-slot worktrees" the old `_dev_repo_prepare` note pointed at ŌĆö now built. Config (declared in `.zshrc`, overridable in `~/.zshrc.local`): `DEV_WORKTREE_ROOT` (default `~/code/.worktrees`, a central dir OUTSIDE every repo's tree so it pollutes nothing), `DEV_WORKTREE_DEFAULT` (global on/off), and the `DEV_WORKTREE` assoc array (per-repo opt-out, keyed like `DEV_BRANCHES`: `DEV_WORKTREE[repo]=0` keeps that repo on the old shared tree). Helpers: `_dev_worktree_enabled <repo>`, `_dev_worktree_path <repo> <slot>` (= `$DEV_WORKTREE_ROOT/<basename>/<slot>`), `_dev_worktree_branch <repo> <slot>` (= `dev/<basename>-<slot>`) ŌĆö both keyed on the repo **basename** (`${DEV_REPOS[repo]:t}`), NOT the alias, so the path + branch are identical on every host (a dir is `dot` here, `dotfiles` there; basename is stable), which keeps cross-host resolution coherent. **`_dev_worktree_create <repo> <slot>`** is the idempotent engine: present worktree ŌåÆ reuse as-is (no re-fetch/re-branch ŌĆö this is what makes reattach free); branch lingers (kill-without-merge) ŌåÆ resume it; fresh slot ŌåÆ `git worktree add -b dev/<basename>-<slot> <path> origin/main` after `git fetch origin`. It prints the path (the caller uses it as the session `dir`) or returns 1 ŌåÆ caller falls back to the shared tree. `_t_dev`/`_t_paste` call it before `_dev_new_session` and pass `skip_prepare=1` (worktree mode), so the in-pane `_dev_repo_prepare` branch dance is **skipped** ŌĆö there is no shared tree and no sibling to trample, so the whole reason `_dev_repo_prepare` existed is gone for worktree repos.
  - **`_dev_repo_prepare` still exists** but runs ONLY for opt-out repos (`DEV_WORKTREE[repo]=0`) and the `-f` inline fallback: tree dirty ŌåÆ switch branch but never stash; tree clean ŌåÆ fetch + checkout + `pull --ff-only`. (Its old full note: it replaced a destructive `git stash; checkout; pull` dance ŌĆö **don't reintroduce `git stash`**, that was the branch-trampling bug.)
  - **Cleanup ŌĆö `_dev_worktree_sweep` (precmd, stamp-gated like `_csync_periodic`).** Every ~600s, detached, it reaps each `$DEV_WORKTREE_ROOT/<basename>/<slot>` worktree whose tmux session is **dead** (matched by `session_path`, never by name ŌĆö dodges alias drift) AND whose branch is **merged** (`_dev_branch_merged`: prefers `gh pr list --head <branch> --state merged` to catch GitHub **squash-merges**, which `git branch --merged`/merge-base miss; an OPEN PR is a hard not-merged; git-only ancestor test is the gh-absent fallback). **Any inconclusive answer = NOT merged**, so unmerged work is never destroyed (a killed-but-unmerged slot keeps its worktree ŌåÆ reopening the slot resumes the branch). Removes the worktree + deletes the branch + prunes. Logs to `~/Library/Logs/dev-worktree-sweep.log`.
  - **`_dev_repo_of_dir <dir>` ŌĆö the single pathŌåÆrepo resolver (zsh + a Python twin `Config.repo_of_dir` in `bin/t`).** Prints `<alias>\t<slot>` for any path: a canonical repo dir/subdir (slot empty) OR a per-session worktree (slot = the path's `<slot>`). Worktree-awareness lives here ONCE; every cwdŌåÆrepo site delegates to it ŌĆö `_dev_cwd_repo_dir`, `_t_infer_repo`, `_dev_slot_for_cwd`, the `_dev_fg_rows` registry mapping, and bin/t's `repo_dir_for_cwd`/`_infer_repo`. Scope filters (`dev ls`, `_dev_list_remote`, bin/t `_scope_filter`) use the sibling **`_dev_dir_in_scope <dir> <scope>`** (or the `wt` awk clause) so a worktree session is listed under its repo, not dropped. `_dev_slot_for_cwd` re-lands a beamed worktree into its own slot number when free.
  - **tbeam cross-host:** `_tbeam_land` and `_dev_pull` materialize the worktree locally via `_dev_worktree_create` (from its branch on origin) before resuming, instead of hard-failing on a missing dir. **Uncommitted worktree edits now ride along** (closing the old gap): **`_dev_worktree_beam_push`** runs ON THE ORIGIN before the move (locally on the SEND path; over ssh via `TB_WT`/`TB_HOST` env on the RECEIVE path in `_dev_pull`) ŌĆö `git add -A` + a throwaway `wip: beam to <host> [skip ci]` commit + `push origin HEAD:<br>` ŌĆö and **`_dev_worktree_beam_sync`** runs ON THE DESTINATION (in `_tbeam_land`/`_dev_pull`, after `_dev_worktree_create`) to **fast-forward ONLY** the worktree to that pushed tip (a fresh worktree is already at the tip = no-op; a reused/stale one is brought up). Both no-op outside a per-session worktree (`$cwd` under `$DEV_WORKTREE_ROOT`), so opt-out/shared-tree repos are untouched and a beam-commit can't sweep a sibling's WIP. Divergence (a destination-local commit never pushed ŌĆö only under manual interference, since a move leaves no live session there) is **warned, never `reset --hard`**, matching the sweep's "don't clobber on doubt". A failed origin push warns but never aborts the move (the WIP commit is safe in git locally, just not yet on the destination). **Load-bearing fix:** the dev-slot SEND path (`_t_beam`, the `repo_arg` branch) now sets `cwd` from the tmux `session_path` (the actual worktree), not `${DEV_REPOS[repo]}` (the canonical dir) ŌĆö the old value pointed the transcript rsync + push + `TB_CWD` at the wrong dir for worktree sessions (only the in-session `cwd=$PWD` path worked). Needs `dots` on the host (shared `_tbeam_land`/`_dev_worktree_beam_*`), same caveat as the rest of tbeam.
  - **Manual resume-through-sync (the picker path, restored).** Worktree-per-session broke a workflow tbeam doesn't cover: pick a csync'd transcript from the **local picker** and resume it (no host move). It broke two ways, both because a dev session's recorded cwd is now its ephemeral worktree path (`$DEV_WORKTREE_ROOT/<basename>/<slot>`) instead of the always-present dev clone: **(1) the picker hid it** ŌĆö `_claude_session_rows`'s scope filter matched only `cwd == $PWD || cwd startswith $PWD/`, so a repo-scoped picker (or one run from inside a worktree) never showed any worktree session (incl. one synced from another machine, where only the branch + transcript travel). Fixed by normalizing the filter to the **canonical repo dir** via `_dev_repo_of_dir` and passing `DEV_WORKTREE_ROOT` into the python so it also matches `ŌĆ”/<repo-basename>/<slot>` ŌĆö the twin of `_dev_dir_in_scope`. **(2) the resume hard-failed** ŌĆö `_t_push`'s `[[ -d $cwd ]]` guard bailed when the worktree dir was absent (other machine, or reaped after merge). Fixed with **`_dev_ensure_session_cwd <cwd>`** (next to `_dev_dir_in_scope`): present dir ŌåÆ echo it; absent **worktree** path ŌåÆ rebuild it from its branch via `_dev_worktree_create` (same engine `_tbeam_land`/`_dev_pull` use) and echo the identical path so `claude -r` lands on the recorded cwd; absent non-recoverable dir (non-DEV, or worktree-opt-out repo) ŌåÆ rc 1 and the caller errors. *Not* fixed (out of scope, can't be): Claude's **native** `claude --resume` picker still keys to the literal `$PWD` project dir, so it won't surface worktree transcripts from the canonical repo dir ŌĆö use the `t push -p` / `_claude_sessions_fzf` picker, which now does.
  - **dotfiles is IN scope** (it's the motivation). Caveat: `dots --dev` still repoints the live symlinks at the **dev clone** (`~/code/dotfiles` on `dev/claude-1`), not at a session worktree, so testing a dotfiles session's edits live still means mergeŌåÆ`dots`.
- **`bin/claude-stamp-tmux`** ŌĆö `set -uo pipefail` Bash run as a **SessionStart hook** (wired in `~/.claude/settings.json` ŌåÆ `hooks.SessionStart`, command `$HOME/bin/claude-stamp-tmux`; the tracked seed is `claude/settings.json.example`). On every session start it stamps that session's id onto its tmux session as `CLAUDE_RESUME_ID` (`tmux set-environment`, which lands on the right session via the inherited `$TMUX_PANE`). **Why it exists:** `tplan`/`tpop`/`_dev_session_summary` resolve a slot's session id from `CLAUDE_RESUME_ID`, and when it's unset they fall back to "newest `.jsonl` by mtime in the project dir" ŌĆö which is **ambiguous when several slots share one repo dir** (`dev-api-1..5` all root at `~/code/my-api`, so one `~/.claude/projects/<enc>/`); the fallback returns whichever *sibling* slot wrote last, not the slot you asked for. `_dev_new_session`/`_dev_resume_session` already stamped the id, but sessions started any other way (bare `claude`, `tmux new-session ŌĆ” send-keys claude`) never got stamped ŌĆö that was the `tplan api 2` "wrong/no plan" bug. The hook closes the gap for *every* launch path. Load-bearing details: reads `session_id` from the hook's **JSON payload on stdin** (via `python3`), falling back to `$CLAUDE_CODE_SESSION_ID`; a **no-op outside tmux** (`[[ -n $TMUX ]]`); **always exits 0** so it can never block a session from starting. Does *not* retroactively fix already-running pre-hook sessions (their id was never recorded and isn't recoverable from the live process ŌĆö Claude doesn't hold the `.jsonl` open) ŌĆö for those, `_dev_session_summary`'s birthtime-matching is the only heuristic, and it still misses when the original transcript has been cleaned up. **Second job ŌĆö the foreground registry (now runs for *every* launch, not just tmux):** the hook also writes one file per live claude pid, `~/.cache/claude-sessions/<pid>` = `<session_id>\t<cwd>` (pid found by walking up from `$PPID` to the nearest `comm == claude`; session_id + cwd from the stdin payload). This is the reliable `pid ŌåÆ session_id` link `dev ls` needs to show **foreground (non-tmux) claudes** with their real transcript title ŌĆö macOS hides another process's env (so nothing can read `CLAUDE_CODE_SESSION_ID` off a running claude) and csync scrambles transcript birthtimes (so the heuristic mis-picks), leaving the registry the only trustworthy source. `_dev_fg_rows` reads it and prunes dead-pid entries; a session started *before* the hook recorded it renders as `(foreground claude)`. Only the `tmux set-environment` stamp is still gated on `$TMUX`.
- **`bin/pii-scan`** ŌĆö `set -euo pipefail` Bash + a Python heredoc that fails if any PII appears in tracked/staged files. Uses a **two-layer ruleset**: (1) the **denylist** `scrub-rules.json` ŌĆö literal, case-sensitive personal identifiers, marked PRIVATE and **never committed here** (gitignored as defense-in-depth); read locally from `~/.config/pii-scan/scrub-rules.json` (override `$PII_RULES`) and in CI from the `PII_SCRUB_RULES` secret; (2) the **ignore patterns** `pii-ignore-patterns.txt` ŌĆö regexes for documented false-positive *shapes* (no PII, tracked). Plus a dotfiles-only third layer: (3) the **allowlist** `pii-allowlist.txt` ŌĆö intentionally-public values for *this* repo (your GitHub handle; generic vendor names like `Anthropic`/`OpenAI` that the shared denylist scrubs elsewhere but are intrinsic to a Claude Code dotfiles repo). A denylist hit clears only if `value in allowEntry and allowEntry in line` (whitelisting a full email also clears its domain substring, but only on lines that contain the full email) or the value matches an ignore regex. **Don't edit the shared denylist to silence a dotfiles false positive ŌĆö add to `pii-allowlist.txt` instead.** Two run modes: `--staged` (staged blobs via `git show :file`, used by the hook) and default (all `git ls-files`). **Fails open when the denylist is absent** (machines without the denylist still commit) unless `--require-rules` (CI) is passed, so a missing secret is loud. Wired two ways: the tracked **`.githooks/pre-commit`** (enabled by `install.sh` via `git config core.hooksPath .githooks`) runs `--staged` locally; the **`pii-scan` job in `.github/workflows/ci.yml`** runs `--require-rules` on push/PR to `main`, materializing the secret to a temp file. Bypass a local block once with `git commit --no-verify`.
- **`tests/` ŌĆö the pytest suite for the Python bins.** Covers the **pure logic** in `bin/t` (config parsing + `_CFG_LINE`, `Config.repo_of_dir`/`repo_dir_for_cwd`, the `_clean_capture`/`_keep_sgr` capture-pane ANSI stripper, `_parse_rows`/`_scope_filter`/`_truncate`/`_infer_repo`) and `bin/pr-watch` (the autonomous-fix triage gate: `classify`/`failing_checks`/`pending_checks`, plus `_load_config_file`/`load_state`/`save_state`/`write_brief`). **Subprocess/ssh/tmux/launchd/CLI glue is out of scope** ŌĆö not unit-tested, and excluded from coverage measurement. **Load-bearing detail: the bins are NOT renamed to `.py`** (that would break the PATH symlinks, the `t()` shim, and the pr-watch plist) ŌĆö `tests/conftest.py` imports them in place via an `importlib` `SourceFileLoader` (`spec_from_file_location` returns `None` for an extensionless path, so the loader is passed explicitly); importing is side-effect-safe because both guard their entrypoint with `if __name__ == "__main__"`. Coverage is **scoped in `pyproject.toml`** to just the two files (`[tool.coverage.run] include`) with the glue excluded by `[tool.coverage.report] exclude_lines` regexes that match the `def`/`class` headers of subprocess/CLI functions (a header match excludes the whole block) ŌĆö so the percentage reflects the tested surface, not diluted by un-unit-testable I/O. Config-only `pyproject.toml` (no `[build-system]`; the repo ships no package) and `requirements-dev.txt` (`pytest`, `pytest-cov`) are **CI/dev-only** ŌĆö never installed by `install.sh`. Tests monkeypatch module globals (`CONFIG`, `STATE_FILE`, `TASKS_DIR`, `HOME`) + `chdir` rather than touching real `$HOME`. Run: `python3 -m pytest`.
- **`.github/workflows/ci.yml` ŌĆö the single-gate CI pattern.** shellcheck + gitleaks + pii-scan + pytest + pip-audit + codeql are jobs in *one* workflow plus a final **`ci` job named `CI`** that `needs:` all of them. (**`pip-audit`** scans `requirements-dev.txt` ŌĆö the only dependency manifest ŌĆö for known CVEs; **`codeql`** runs SAST over the Python only, finding the extensionless `bin/t`/`bin/pr-watch` via their shebang, with job-level `security-events: write` to upload SARIF ŌĆö don't add a `.py`-only paths filter, it would drop the bins.) **Branch protection requires exactly one check: `CI`** ŌĆö uniform across every repo, so there is nothing per-repo to remember (`needs:` only works within a single file, which is why they were merged). The gate is **`if: always()`** (without it the gate is *skipped* when a dependency fails, so the required check never reports failure and the PR merges anyway) and loops over `needs.*.result`, treating **`skipped` as a pass** so the self-skipping `pii-scan` (fork/Dependabot PRs) does not block them. Add/remove jobs freely ŌĆö just keep them in the gate's `needs:` and keep the gate named `CI`. **The `pytest` job** runs `pytest --cov --cov-fail-under=95` (3.12, `pip install -r requirements-dev.txt`); the threshold is a **ratchet** ŌĆö raise it as coverage grows, never lower it (set a few points below the measured value so a minor refactor does not spuriously fail). **`scorecard.yml` stays separate and is NOT in the gate:** it has no `pull_request` trigger and is public-gated, so it never produces a PR check (requiring it would wedge PRs at "Expected ŌĆö waiting for status").
- **`claude/settings.json.example`** ŌĆö the only tracked Claude settings file: the seed for the real copy at `~/.claude/settings.json`, carrying just the `SessionStart` ŌåÆ `claude-stamp-tmux` hook (see its note above).
