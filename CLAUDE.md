# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Workflow

`main` is protected ‚Äî it takes changes through PRs, not direct pushes. Do work on the `dev/claude-1` branch (the repo's standing dev branch; `t open dotfiles` checks it out by default now that the `DEV_BRANCHES[dotfiles]=main` override is gone). Commit and push to that branch as you go ‚Äî no need to ask first ‚Äî and open a PR to `main` when a change is ready. The symlink model still means edits take effect immediately on disk; the PR is just how they land on `main`.

## What this is

Personal macOS dotfiles: zsh config, utility scripts, and Claude Code config. No build system, no tests, no dependencies ‚Äî just shell files and a Bash install script.

## The symlink model (read this first)

The files **in this repo are the source of truth**. `install.sh` creates symlinks from `$HOME` back into the repo, so the home-directory copies and the repo copies are literally the same file:

| Repo file | Symlinked to |
| --- | --- |
| `.zshrc` | `~/.zshrc` |
| `bin/sleep-manager` | `~/bin/sleep-manager` |

Consequences:
- Editing either side edits both. There is no "sync" or "copy back" step.
- `install.sh` is location-independent (it resolves its own dir via `BASH_SOURCE`), so the repo can live anywhere; re-run it after moving the repo to relink.
- Running `install.sh` backs up any real file in the way to `<name>.bak` before linking, and `rm`s a pre-existing symlink. Adding a new managed file means adding a `link` call in `install.sh`.
- **Deliberate exceptions to the symlink model:** `~/.zshrc.local` is a *real copy* (seeded by `install.sh` from `.zshrc.local.example`), not a symlink, so per-machine/private config (real `DEV_REPOS` paths, `TBEAM_HOST`, private completions) stays out of the repo. `.zshrc` sources it if present. Never commit a real `~/.zshrc.local`. **`~/.claude/settings.json`** is the same pattern ‚Äî a real copy seeded from `claude/settings.json.example` (hook only), *always*, because Claude Code **writes** to the live file at runtime (`/model` saves the default model, "always allow" appends permission rules, plugin toggles land there); a tracked/symlinked copy kept the repo dirty and risked committing private allow-rules, so the old author-mode symlink (install choice 2 / `CLAUDE_SETTINGS=author`) was dropped ‚Äî `install.sh` materializes a legacy symlink into a real copy of its current content. An existing settings file is never clobbered, and `claude/settings.json` is gitignored as defense-in-depth.

## Commands

```sh
./install.sh            # link all dotfiles into $HOME (idempotent; re-run to relink)
t ls [-r] [-a]          # list Claude dev sessions (-r cross-host, -a all repos)
t open [repo] [slot]    # open/reattach (repo defaults to cwd repo; --new, --fg, --remote); auto-attaches a remote-only slot in place
t beam [repo] [slot] --from <host>   # pull a session from another machine to here (receive); omit --from to send
t push | t pop          # move this session into / out of a detached tmux slot
t beam <repo> [slot] --host mini   # send a session to another machine
sleep-manager status    # show pmset timeouts, active sleep assertions, script-managed caffeinate
sleep-manager disable   # block sleep (sudo): pmset disablesleep 1 + backgrounded caffeinate
sleep-manager enable    # restore sleep (sudo): kill caffeinate + reset pmset defaults
```

There is no lint/test step. After editing a shell file, validate by sourcing it (`source ~/.zshrc`) or running the script. For `bin/t`: `python3 -m py_compile bin/t` (write the `.pyc` to /tmp so it isn't committed ‚Äî `__pycache__/` is gitignored).

## Component notes

- **`bin/t` + the `t()` shim ‚Äî the single Claude-session command (READ THIS FIRST for the session tooling).** The whole `dev`/`tpush`/`tpop`/`tbeam`/`tread`/`tplan`/`tpaste`/`tfind`/`on` family was unified into one **gh-style** command `t <verb> [args] [--flags]`. Architecture is **bin core + thin zsh shim** (a pure bin is impossible for some verbs ‚Äî see below):
  - **`bin/t`** (Python; symlinked onto PATH by `install.sh`, present on every host) owns argument parsing + per-verb `-h` help (argparse), the config parser, the cross-host rendering, and the subprocess-safe verbs. It implements `ls` (renders natively in Python, reusing the proven `_dev_*` data via `zsh -lic` ‚Äî output matches the old `dev ls` row-for-row), `read` (log pager, native), `on` (native; two-layer quoting + `zsh -lic` contract), and delegates `plan`/`paste`/`kill` to the `_t_*` zsh helpers. Internal `session-rows`/`land`/`kill-owner` expose the remote contract.
  - **`t()` in `.zshrc`** is the front door. Bin-native verbs run straight through via **`command t`** (the `command` builtin reaches `~/bin/t` past the function ‚Äî the same name-collision trick `claude()` uses). The **shell-bound** verbs (`open`/`pop`/`push`/`find`) map to the existing zsh helpers, which already do the `cd` + `claude -r` in the *current terminal* and the tpush sentinel handoff correctly **because they run in the calling shell** ‚Äî a bin subprocess can't. So there is **no resolve protocol**; the shim just calls them, and the one-live-owner invariant is literally still `_t_push`/`claude()` unchanged.
  - **Hard cutover done:** the 9 old top-level names are **gone** (typing `dev`/`tpush`/‚Ä¶ ‚Üí command not found). Their implementations were renamed to internal helpers ‚Äî **`dev‚Üí_t_dev`, `tpush‚Üí_t_push`, `tpop‚Üí_t_pop`, `tbeam‚Üí_t_beam`, `tplan‚Üí_t_plan`, `tpaste‚Üí_t_paste`, `tfind‚Üí_t_find`** ‚Äî and `tread`/`on` were reimplemented natively in `bin/t`. **The detailed notes below still describe this logic and its invariants accurately; only the function names + how they are invoked changed.** Grammar mapping: `dev‚Üít open`, `dev <repo> new‚Üít open <repo> --new`, `dev ‚Ä¶ -f‚Üít open ‚Ä¶ --fg`, `dev ls‚Üít ls`, `dev kill‚Üít kill`, `dev -r‚Üít open --remote` (now also auto-detected ‚Äî a slot live only on another host is attached in place without the flag), `dev -r ‚Ä¶ --here‚Üít beam <repo> <slot> --from <host>` (the pull/receive moved to tbeam), `dev -r kill‚Üít kill --remote`, `dev <repo> fg‚Üít open <repo> fg` (the separate `t fg` verb was folded into `t open`: `t open <id>` / `t open <repo> <id>` / `t open <repo> fg` all adopt a foreground session here ‚Äî an id-shaped arg, 4+ hex chars or the displayed `repo:id` label, dispatches via `_dev_fg_handle`; pure digits stay slot numbers), `tpush‚Üít push`, `tpop‚Üít pop`, `tbeam [host]‚Üít beam --host <h>`, `tread‚Üít read`, `tplan‚Üít plan`, `tpaste‚Üít paste`, `tfind‚Üít find`, `on‚Üít on`.
  - **Repo-aware positionals (every slot verb):** any verb taking `<repo> [slot]` infers the repo from `$PWD` when called bare or with a lone numeric slot ‚Äî `t paste 4` in `~/code/financial-forecast` pastes into *that* repo's slot 4; bare `t open` opens/reattaches the repo you're standing in; likewise `t open 4`/`--new`, `t kill 4`/`t kill all`/bare `t kill` (infers the repo, still demands a slot), `t pop 4` (and bare `t pop` now matches from a subdir ‚Äî repo dir, not exact `$PWD`), `t plan 4`, `t read 4`/bare `t read`, `t beam 4 [--host h]`/`t beam [4] --from <host>`, and no-arg fg adoption (`t open fg`) prefers this repo's foreground claudes. Inference is **`_t_infer_repo [slot]`** (.zshrc, next to `_dev_cwd_repo_dir`): a **live `dev-*` session rooted in the cwd's repo dir wins** ‚Äî exact slot's session first, then any live one ‚Äî with the alias read off the actual session *name*, so it matches however the slot was created when several aliases key one dir (`dev-dot-4` vs `dev-dotfiles-4`); no live session ‚Üí the dir's DEV_REPOS key (basename match, then shortest). `t read` is bin-native, so `bin/t` has the twin **`_infer_repo`** working over the log files instead (exact-slot log, then any `dev-<alias>-N.log`, **newest mtime wins** ‚Äî logs outlive sessions, so stale-alias logs lose to the alias in use). Deliberately **not** inferred: the bare `t open -r` every-remote-slot picker (documented behavior), and `t beam --from` uses alias-only inference (the slot lives on the *remote* host, so a same-numbered local session name is no evidence).
  - **Config bridge:** `bin/t` can't read zsh assoc arrays, so `_t_sync_config` (in `.zshrc`) derives **`~/.config/t/config.sh`** (plain `NAME[key]=value` lines) from the live `DEV_REPOS`/`DEV_BRANCHES`/`REMOTE_HOSTS`/`DEV_BRANCH`, mtime-gated; `bin/t` parses it with a strict regex (never sources it; `~/.zshrc.local` stays the source of truth and is never executed by the bin). zsh keeps using the live arrays for the cd/host shortcuts + the `_t` completion.
  - **Remote contract unchanged:** the cross-host fan-out still runs `_dev_session_rows`/`_dev_rows_all`/`_tbeam_land`/`_tbeam_kill_owner` over `ssh ‚Ä¶ zsh -lic` ‚Äî those are `_`-prefixed internals that **survive the cutover** (the `zsh -lic` is required because `~/bin` + Homebrew are only on the interactive PATH). Remote `_dev_remote` attach/kill now run `t open`/`t kill` on the far host (which has only `t`). **Adding `bin/t` to a host needs `install.sh` there, not just `dots`** (`dots` pulls + reloads but does not relink a *new* bin script); the rc-127 "stale dotfiles" note in the fan-out is the in-band reminder. **Minor known gap:** `t on` does not set the Terminal tab title (the old `on` called `_term_title`) ‚Äî a cosmetic follow-up.
- **`bin/csync`** ‚Äî `set -euo pipefail` Bash that two-way-merges Claude session history between `~/.claude/projects` and the iCloud Drive `claude-sessions` folder (`rsync -a --update`, no `--delete`, so machines converge to the union). Was a `.zshrc` function; extracted to a script (`help` still lists it ‚Äî it scans `~/bin`). `brctl download` is wrapped in `|| true` so its exit status never aborts the sync under `set -e`. **Periodic runs come from a `precmd` hook in `.zshrc` (`_csync_periodic`), not launchd** ‚Äî this is load-bearing and a dead end was hit getting there: iCloud Drive is TCC-protected and *background launchd agents are denied*, and on recent macOS (Tahoe) granting `/bin/bash` Full Disk Access has **no effect** (verified ‚Äî even a pure-bash `readdir` is denied under a launchd agent), because TCC won't pin an FDA grant to a platform interpreter running an arbitrary script. The shell hook sidesteps this entirely by running `csync` in the Terminal's already-approved context. The hook gates on a `~/.cache/csync-last-run` stamp (written *before* the run so overlapping shells don't double-fire), uses `$EPOCHSECONDS` (no `date` fork per prompt), and launches `csync` detached (`( csync ‚Ä¶ & )`) so it never blocks the prompt; background runs log to `~/Library/Logs/csync.log`. Don't reintroduce a launchd agent for this without a code-signed helper binary.
- **`bin/sleep-manager`** ‚Äî `set -euo pipefail` Bash with a `case` dispatcher. Sleep blocking is two layers: `pmset -a disablesleep 1` (OS-level, survives logout/reboot) plus a backgrounded `caffeinate -dimsu` whose PID is tracked in `/tmp/sleep-manager-caffeinate.pid`. Key gotcha documented in its `--help`: the caffeinate process dies on reboot but the pmset flag does **not** ‚Äî always run `enable` to fully restore; don't rely on a reboot. `status` is the source of truth for current state.
- **`.zshrc`** ‚Äî PATH (`~/bin` first), then `compinit` is run **before** sourcing anything that calls `compdef` (e.g. a completion sourced from `~/.zshrc.local`) ‚Äî order matters, this was a fix for a "command not found: compdef" error. Also defines `prview` (gh PR check summary via jq) and the interactive `nosleep` one-liner (the inline cousin of `sleep-manager`). **Per-machine config is split out:** `.zshrc` declares an empty `DEV_REPOS` and sources `~/.zshrc.local` (a real copy, not symlinked ‚Äî see the symlink-model note), which holds the real repo paths, `export TBEAM_HOST`, and any private completions. The committed `.zshrc` ships generic examples and derives all usage/error/completion strings from `${(k)DEV_REPOS}`, so it works (and stays generic) even with no local file. `install.sh` seeds `~/.zshrc.local` from `.zshrc.local.example`. **Help is single-source-of-truth, split into extractor + renderer:** the doc-comment block directly above a `name()` function *is* that command's `-h`/`--help` text. **`_help_for name`** `awk`-extracts the contiguous `#` block immediately above `name() {` (a **truly blank** line resets it; a bare `#` is kept as a spacer, so rationale can sit above the help separated by one empty line), strips the `# `, and pipes it through **`_help_style`** ‚Äî the standalone gh-style RENDERER (split out of `_help_for` so the dynamic paths below can reuse it). `_help_style` reads plain text on stdin and styles a gh-style convention: **line 1** `name ‚Äî tagline` (name bolded on the ` ‚Äî `), a `Usage:` line (label bold; an indented *second* Usage line stays verbatim under it), bare `Word:` section headers (`Commands:`/`Options:`/`Arguments:`), and **2-space-indented `term  <2+sp>desc` rows** (term cyan, desc dim, descriptions **auto-aligned** to the widest term so comment spacing needn't be hand-tuned). A **wrapped description line** ‚Äî indented **3+** spaces right after a kv row (or its own continuation), tracked by an `eligible` flag so a Usage/paragraph follow-on is excluded ‚Äî is hang-indented to the same description column (this is why a row's continuation comment no longer needs hand-alignment; just indent it). Color is gated on `[[ -t 1 ]]` in `_help_style`, which is the **last** stage of every help pipe, so the test sees the real terminal and `cmd -h | cat` stays plain. To add/restyle a command's help, just write its comment in that shape ‚Äî no per-command formatting code. **The dynamic no-arg/bad-arg paths (`dev`, `dev kill`, `tread`) now build their text ‚Äî including the *actual* `${(k)DEV_REPOS}` / live logs that static help can't show ‚Äî and pipe it through `_help_style` too, so they render identically to `-h`** (no more ad-hoc `echo "Usage: ‚Ä¶"`). **Gotcha:** `_help_style`'s awk body is a single-quoted string, so **no apostrophes in its inline comments** (`is NOT`, not `isn't`) ‚Äî one closes the quote and breaks the source.
- **`tfind` (in `.zshrc`)** ‚Äî semantic search across saved Claude sessions: "which session was working on X?". Pipeline is **retrieve-then-rerank** ‚Äî a keyword pass over each transcript's title + your prompts builds a candidate pool (the same scan `_claude_sessions_fzf` uses; padded with recent sessions when keyword matches are thin, so divergent wording still reaches the model), then **Sonnet via `claude -p --model sonnet --output-format json`** reranks the candidates and returns a JSON `[{i, why}]` ranking. Two load-bearing details: (1) the model call is a **Python `subprocess.run(['claude', ‚Ä¶])`**, which execs the real `~/.local/bin/claude` binary and so **bypasses the zsh `claude` wrapper** (the wrapper's tpush sentinel would otherwise interfere); (2) **every failure path ‚Äî no CLI, timeout, unparseable reply, empty `[]` ‚Äî falls back to keyword order** so the fzf picker always populates and the command works offline. `tfind -k` forces the offline keyword path (no Sonnet). The keyword scorer (`hay.count(t) + 4*head.count(t)`) is recall-only here ‚Äî Sonnet does the precision ‚Äî so don't over-tune it.
- **`tbeam` (in `.zshrc`)** ‚Äî **moves** a Claude session between machines; a **cross-host `tpush`/`tpop`**. **Both directions live here:** by default it **sends** the resolved session to a host (default `$TBEAM_HOST`, set in `~/.zshrc.local`) and resumes it there; **`tbeam <repo> [slot] --from <host>`** **receives** ‚Äî pulls a session that lives on `<host>` down into a local dev slot (the `--from` branch is parsed early in `_t_beam` and delegates to **`_dev_pull`**, which was the engine behind the old `dev -r ‚Ä¶ --here`; repo is required, slot optional ‚Üí the host's first live slot). This is the home of all session *moving*; merely *attaching* a remote slot in place (no move) is `t open`'s auto-detect, not tbeam. The SEND path resolves a session exactly like `tpush` (current chat via `$CLAUDE_CODE_SESSION_ID`, else fzf-pick scoped to `$PWD`, `-a` for all; **`-s`/`--session <id>`** targets an explicit id/unique-prefix, resolved against local transcripts by globbing `~/.claude/projects/*/<id>*.jsonl` for a unique match then `_tbeam_transcript_cwd` for its recorded cwd), `rsync`s its `~/.claude/projects/<enc>/` dir over (`_tbeam_sync_transcript`, `--update`/no-`--delete` so the live source wins but a newer remote is never clobbered ‚Äî csync still reconciles in the background), then has the **host land it**. Landing logic (`_tbeam_land`) lives in the *shared* dotfiles so it runs on the far side via `ssh "$host" "TB_CWD=‚Ä¶ TB_SID=‚Ä¶ TB_MODE=‚Ä¶ zsh -lic _tbeam_land"` ‚Äî work passed in **`TB_*` env vars** (sidesteps nested-quote hell) and a **login+interactive** shell (`zsh -lic`) because Homebrew `tmux`/`fzf` are on the *login* PATH (`.zprofile`) while `claude` is on the *interactive* PATH ‚Äî plain `ssh host '‚Ä¶'` sees neither. Default mode reuses the host's own `_dev_resume_session` to make a first-class `dev-<repo>-<slot>` (so `dev`/`tread`/`tpop` work there), then auto-attaches via `ssh -t`; `-f` resumes in the ssh foreground, `-d` leaves it detached with an attach hint (`t open <repo> <slot>` to attach it in place, or `t beam <repo> <slot> --from <host>` to pull it back). Two load-bearing constraints: (1) the repo must exist at the **same path** on the host (all repos live in `~/code` on every machine) ‚Äî `tbeam` guards with `ssh host test -d`; (2) from *inside* Claude it forces `-d` (a Bash subprocess has no TTY to `ssh -t` into). `_tbeam_land` is shared code, so after adding/changing it run `dots` on the host to pull it. **tbeam is a move, not a copy:** once the session lands on the far side, the origin's live owner is *stopped* so exactly one live claude owns the id (two live owners of one transcript have no lock and diverge). The SIGTERM-self step gates on **`self_move`** (`sid == $CLAUDE_CODE_SESSION_ID`), not merely "inside Claude" ‚Äî so `tbeam -s <other-id>` run from inside a Claude session stops *that id's* dev slot via the shared `_tbeam_kill_owner` and never kills the foreground claude you're typing in; a true current-session move SIGTERMs the foreground claude after landing (mirrors `tpush`'s auto-exit). `_tbeam_kill_owner` is shared too (also used by the `--from` receive in `_dev_pull`), another reason the move needs `dots` on the host. (There is no `--no-tmux` flag ‚Äî `-f` is the no-tmux mode: foreground resume, no dev slot.) To *survey* what's live across machines before moving, see **`t ls -r`**; **`t open <repo> <slot>`** attaches one in place (auto-finding its host) and **`t beam <repo> <slot> --from <host>`** pulls it down here.
- **`t open` remote attach ‚Äî auto-detect + explicit `-r` (in `.zshrc`, `_t_dev`)** ‚Äî `t open` is the one "get into a session" verb, local or remote, and **it never moves a session ‚Äî it attaches in place wherever the slot lives** (moving is `tbeam`'s job). **The repo arg is optional** ‚Äî bare `t open` in a `DEV_REPOS` dir defaults to that repo via `_t_infer_repo` (see the repo-aware positionals note above; the merge dropped main‚Äôs simpler `_dev_cwd_repo_key` twin); a lone slot (`t open 2`/`t open fg`/`t open new`) with no valid repo likewise targets the cwd repo. **Ordering matters:** the auto-detect probe runs *after* this repo-resolution (and after fg-adoption returns), so even a bare `t open` resolves its repo first and can then find a slot that is live only on another host ‚Äî the probe sat before repo-defaulting in the original commit and was moved down when it merged with the cwd-default work. Resolution order in `_t_dev`: a **live LOCAL slot always wins** (cheap `_dev_local_slot_live` tmux check, no ssh); only when the named slot is **not live here** does it probe `$REMOTE_HOSTS` (host auto-inferred via `_dev_remote_resolve`/`_dev_rows_all`) ‚Äî live there ‚Üí **attach in place** over `ssh -t <target> "zsh -lic 't open <repo> <slot>'"` (the session stays put, the old `tgo`); live nowhere ‚Üí fall through to the **local fresh-start** path. The **auto-probe is gated** to keep the common path cheap: skipped when no `$REMOTE_HOSTS` are configured (free on a single-host box), for an explicit fresh/foreground request (`new`/`fg`/`-f`), and outside `DEV_REPOS` ‚Äî and it only runs at all when nothing is live locally for that repo/slot. **`-r`/`--remote` is the EXPLICIT branch** (`_dev_remote`): force the remote action even when a local slot exists, and the only way to reach the **bare `t open -r` ‚Üí every remote slot** picker (`t open -r api 5` ‚Üí that one, `t open -r api` ‚Üí every `api` slot). One candidate ‚Üí use it; several ‚Üí **fzf-pick** (`host/slot  summary`, needs TTY+fzf); none ‚Üí bail with a `t ls -r` hint. The chosen slot name (`<repo>-<num>`, split on the *last* dash so multi-dash repos like `dotfiles` resolve) gives repo+slot. A no-TTY guard prints the command instead of failing. **Starting a NEW session on a specific host is `t open <repo> [slot] --host <h>`** (`_dev_remote_open`, dispatched from the `_t_open` shim before `_t_dev` ever runs locally): unlike `-r`/auto-detect ‚Äî which only reach an *already-live* slot ‚Äî `--host` forces the host and so can spawn a fresh remote session (`t open ff --host mini`). It forwards the original gh-style args (minus `--host`) over `ssh -t <target> "zsh -lic 't open ‚Ä¶'"`, so the far host's own `t open` re-parses them; no-TTY guard, session stays on `<host>`. **Pulling a remote session HERE is no longer an `open` flag ‚Äî it is `t beam <repo> <slot> --from <host>`** (the receive direction, `_dev_pull`; see the tbeam note). **`t kill -r <repo> [slot]`** tears a slot down on its host ‚Äî resolve ‚Üí `ssh -t <target> "zsh -lic 't kill <repo> <slot> [-y]'"` (remote confirm works through the TTY unless `-y`). `_dev_remote_resolve` (prints `host\trepo\tslot`) is shared by the explicit `-r` attach, the auto-detect attach, and `kill`; `_dev_remote_attach` (resolve once, then ssh -t attach) is shared by the explicit and auto paths so the slot is scanned only once. Because hosts run `_dev_session_rows`/`_tbeam_kill_owner` over ssh, remote `t open` needs their dotfiles current ‚Äî run `dots` there (same caveat as `t ls -r`). **Remote-aware read verbs:** `t plan` (and any future per-slot reader) is host-bound because its artifact is host-local ‚Äî the plan `.md` lives in the slot's `~/.claude/plans` (csync syncs transcripts, *not* plan files), so `t plan ff 1` on a slot beamed to mini must render *on mini*. `_t_plan`'s "No such session" path now calls the shared **`_dev_remote_delegate <repo> <slot> <verb> [extra‚Ä¶]`** (next to `_dev_remote_attach`): if the slot is not live locally but resolves to a remote host, it runs `t <verb> <repo> <slot>` there over `ssh -t` and returns 0; else returns 1 and the caller falls back to its local path. Same gating as the auto-attach (no `$REMOTE_HOSTS` ‚Üí no-op; no-TTY ‚Üí print a hint and still return 0 so it does not fall through to a local error). `t read` (the tmux-log pager, also host-local) is a natural next caller but is not wired yet.
- **`dev ls -r` / cross-host `dev ls` (in `.zshrc`)** ‚Äî `dev list`/`dev ls` with `-r`/`--remote` renders `_dev_list` across **this machine AND every `$REMOTE_HOSTS` host** in one table (same `‚óè attached ¬∑ ‚úì active context` header) with a dedicated **HOST column** (STATUS/HOST/SESSION/WORKING ON) ‚Äî local rows read `local`, remote rows their `$REMOTE_HOSTS` key. It's read-only ‚Äî a survey of "what's running on macbook vs mini"; you *attach* a remote one in place with `t open <repo> <slot>` (host auto-inferred ‚Äî no flag needed) or *pull* it down with `t beam <repo> <slot> --from <host>` (which enforces the move + one-live-owner invariant). **The key design call is *why it lists live dev slots, not transcripts*:** transcripts already converge on every machine via csync, so an all-transcripts cross-host view would be mostly duplicate rows (the same id once per host) ‚Äî whereas live `dev-<repo>-<slot>` slots genuinely differ per machine. Three helpers: **`_dev_session_rows`** is the machine-readable sibling of `_dev_list` (one `sid\tcwd\tslot\tstate\tcontext\tsummary` row per live slot, reusing `_dev_session_has_claude`/`_dev_session_at_welcome`/`_dev_session_summary`; an unstamped/idle slot's empty `sid` is sentinel'd `-` because **tab is an IFS-whitespace char**, so a leading/consecutive empty field would otherwise collapse under `read` and slide the columns); **`_dev_rows_all`** fans it out over `local` + every `$REMOTE_HOSTS` value **in parallel** (background ssh ‚Üí temp files ‚Üí `wait`) with `ConnectTimeout=3`/`BatchMode=yes`, stashing each host's ssh exit code in a `.rc` file to distinguish **unreachable** (rc 255 ‚Üí "unreachable") from **reachable-but-stale-dotfiles** (rc 127 ‚Üí "run `dots` there"), both skipped + noted on stderr so one sleeping Mac never stalls the table ‚Äî then prefixes each row with its host **verbatim** (no re-split, so empty fields still can't collapse); **`_dev_list_remote`** re-renders those rows in `_dev_list`'s format under a HOST column (`$COLUMNS`-truncating WORKING ON). Because each host runs `_dev_session_rows` over ssh, `dev ls -r` needs every host's dotfiles current ‚Äî run `dots` there (same caveat as `dev -r`); the rc-127 "stale dotfiles" note is the in-band reminder. **Scoped by default:** both `dev ls` and `dev ls -r` show only the repo you're standing in ‚Äî `_dev_cwd_repo_dir` maps `$PWD` to its `DEV_REPOS` dir (path-based, so `dot-*` and `dotfiles-*` both show in `~/code/dotfiles`) and the renderers filter rows by that dir; `-a`/`--all` widens to every repo, and standing outside any repo is unscoped. **Foreground sessions too:** `_dev_fg_rows` adds claudes you ran directly in a terminal (live `comm == claude` not in a `dev-*` tmux pane, minus the one *this* shell runs under), labeled `<repo>:fg`, with sid+cwd from the claude-stamp-tmux registry (see that note) so the title is exact; it's appended to `_dev_session_rows` (so `dev ls -r` covers each host's foreground claudes) and rendered by `_dev_list`.
- **`on` + `REMOTE_HOSTS` (in `.zshrc`)** ‚Äî `on <host> [cmd‚Ä¶]` runs a command on another machine over `ssh -t "$host" "zsh -lic <cmd>"`. The **`zsh -lic`** (login+interactive) is the same trick `_tbeam_land` relies on: Homebrew `tmux`/`fzf` are on the *login* PATH while `claude` and the dotfiles functions are on the *interactive* PATH, so a plain `ssh host '‚Ä¶'` sees neither ‚Äî `-lic` makes `dev`/`tread`/etc. resolve remotely (e.g. `on mini dev dot` starts a dev session there). No command ‚áí an interactive shell on the host. The **command quoting is two layers** (`local cmd="${(j: :)${(@q)@}}"` then `ssh ‚Ä¶ "zsh -lic ${(q)cmd}"`): `(@q)` quotes each arg so the remote `zsh -c` sees the original words, then `(q)` wraps the joined string as **one** token so ssh ‚Äî which re-splits its remote command on spaces ‚Äî transports it intact (verified: a `;` in an arg arrives as data, not a separator). Hosts live in **`REMOTE_HOSTS`** (assoc array, declared by `.zshrc`, populated in `~/.zshrc.local` ‚Äî same single-source-of-truth pattern as `DEV_REPOS`): each key auto-generates a **shorthand function** (`mini ‚Ä¶` ‚â° `on mini ‚Ä¶`, made via `functions[$key]=‚Ä¶` in a loop next to the `DEV_REPOS` cd-shortcut loop) plus `_on_hosts` tab-completion and a synthesized `help` entry (the generated functions aren't real text in the file, so ‚Äî like the cd shortcuts ‚Äî `help` synthesizes their entries from the array). An **unknown host is used as a literal ssh target** (`${REMOTE_HOSTS[$h]:-$h}`), so `on box.local uptime` works unregistered. **Back-compat seed:** an old `MINI_HOST`/`TBEAM_HOST` populates `REMOTE_HOSTS[mini]` when the registry has no `mini`, so a `mini` shortcut keeps working with no config change. (`on` is defined far below the generation loop, but function bodies bind late, so a generated `mini` calling `on` is fine.)
- **`tpush`/`tpop`/`claude()` wrapper (in `.zshrc`)** ‚Äî move a Claude session between a foreground terminal and a detached `dev-<repo>-<slot>` tmux session. **The load-bearing invariant: never two live `claude` processes on one session id.** Claude Code takes *no lock* on a session's `.jsonl` and does *not* auto-fork ‚Äî two live resumers of the same id both append with no coordination and the conversation diverges (the copy you aren't driving freezes). So: (1) `tpush` from *inside* Claude does **not** spawn `claude -r` itself ‚Äî it writes a `SPAWN\t<session>\t<cwd>\t<sid>` instruction to the `$CLAUDE_TPUSH_ATTACH` sentinel, and the `claude()` wrapper spawns the detached resume **after** the foreground exits (when the id is free) and then attaches. **The exit is now automatic:** after writing the sentinel, `tpush` walks the process ancestry (`_tpush_claude_pid`) to the nearest `claude` and `kill -TERM`s it, so you no longer type `/exit` ‚Äî SIGTERM exits Claude cleanly in ~2s (transcript is appended live, so only the in-flight tpush turn is lost; terminal is restored on the wrapper's `tmux attach`). The kill fires *after* the sentinel is written and *before* any spawn, so the one-live-owner invariant holds; if the ancestor walk fails it falls back to printing the manual `/exit` hint. **Don't reintroduce a pre-spawn here** ‚Äî the kill-then-wrapper-spawn ordering is what keeps a single owner. (`ATTACH\t<session>` is the already-backgrounded case; a plain-shell picker spawns inline since the picked session isn't live; an older shell with no wrapper spawns eagerly + warns.) (2) `tpop` captures the tmux session's claude PID (pane shell's child), `kill-session`s, then **waits for that PID to actually exit** (‚â§5s) before `claude -r` in the foreground ‚Äî `kill-session` only SIGHUPs, so resuming immediately raced the dying process. Was the "popped/backgrounded session stops updating while the original keeps going" bug. **`dev <repo> <slot> -f` delegates here:** in `dev`, `-f` (no-tmux) foreground-RESUMES the named slot if it's live (‚â° `tpop <repo> <slot>`), and only starts a *fresh* inline claude when that slot doesn't exist ‚Äî so `-f` means "foreground" consistently across `dev`/`tbeam`/`dev -r`, not "fresh" in one and "resume" in the others.
- **`_dev_repo_prepare` ‚Äî new-session branch sync that does NOT trample siblings (in `.zshrc`).** Every `dev-<repo>-*` slot cd's into the **same working tree** (`~/code/<repo>`), so the old fresh-session git dance ‚Äî `git stash; git fetch; git checkout $branch; git pull` ‚Äî was destructive: `git stash` silently pocketed a *sibling* session's uncommitted work, and a default-merge `git pull` could move the branch out from under a session mid-conversation. Both `_dev_new_session` (the tmux send-keys line) and the `-f` inline path now call **`_dev_repo_prepare <branch>`** instead. Policy: **tree dirty ‚Üí do nothing** (leave the checkout exactly as is; never stash ‚Äî a sibling likely owns those changes; print a `‚Ü∑ branch sync skipped` note); **tree clean ‚Üí fetch + checkout + `git pull --ff-only`** (non-destructive; no sibling has local edits to lose; matches the global `pull.ff=only`). It runs in the session's own shell (cwd = the repo, from tmux `-c`/the `-f` `cd`), and is defined in `.zshrc` so the tmux send-keys shell (which sources it) and the `-f` caller both have it. **Don't reintroduce `git stash` here** ‚Äî that was the branch-trampling bug. The deeper fix would be per-slot git worktrees; this is the pragmatic "never destructive on a shared tree" guard.
- **`bin/claude-stamp-tmux`** ‚Äî `set -uo pipefail` Bash run as a **SessionStart hook** (wired in `~/.claude/settings.json` ‚Üí `hooks.SessionStart`, command `$HOME/bin/claude-stamp-tmux`; the tracked seed is `claude/settings.json.example`). On every session start it stamps that session's id onto its tmux session as `CLAUDE_RESUME_ID` (`tmux set-environment`, which lands on the right session via the inherited `$TMUX_PANE`). **Why it exists:** `tplan`/`tpop`/`_dev_session_summary` resolve a slot's session id from `CLAUDE_RESUME_ID`, and when it's unset they fall back to "newest `.jsonl` by mtime in the project dir" ‚Äî which is **ambiguous when several slots share one repo dir** (`dev-api-1..5` all root at `~/code/my-api`, so one `~/.claude/projects/<enc>/`); the fallback returns whichever *sibling* slot wrote last, not the slot you asked for. `_dev_new_session`/`_dev_resume_session` already stamped the id, but sessions started any other way (bare `claude`, `tmux new-session ‚Ä¶ send-keys claude`) never got stamped ‚Äî that was the `tplan api 2` "wrong/no plan" bug. The hook closes the gap for *every* launch path. Load-bearing details: reads `session_id` from the hook's **JSON payload on stdin** (via `python3`), falling back to `$CLAUDE_CODE_SESSION_ID`; a **no-op outside tmux** (`[[ -n $TMUX ]]`); **always exits 0** so it can never block a session from starting. Does *not* retroactively fix already-running pre-hook sessions (their id was never recorded and isn't recoverable from the live process ‚Äî Claude doesn't hold the `.jsonl` open) ‚Äî for those, `_dev_session_summary`'s birthtime-matching is the only heuristic, and it still misses when the original transcript has been cleaned up. **Second job ‚Äî the foreground registry (now runs for *every* launch, not just tmux):** the hook also writes one file per live claude pid, `~/.cache/claude-sessions/<pid>` = `<session_id>\t<cwd>` (pid found by walking up from `$PPID` to the nearest `comm == claude`; session_id + cwd from the stdin payload). This is the reliable `pid ‚Üí session_id` link `dev ls` needs to show **foreground (non-tmux) claudes** with their real transcript title ‚Äî macOS hides another process's env (so nothing can read `CLAUDE_CODE_SESSION_ID` off a running claude) and csync scrambles transcript birthtimes (so the heuristic mis-picks), leaving the registry the only trustworthy source. `_dev_fg_rows` reads it and prunes dead-pid entries; a session started *before* the hook recorded it renders as `(foreground claude)`. Only the `tmux set-environment` stamp is still gated on `$TMUX`.
- **`bin/pii-scan`** ‚Äî `set -euo pipefail` Bash + a Python heredoc that fails if any PII appears in tracked/staged files. Uses a **two-layer ruleset**: (1) the **denylist** `scrub-rules.json` ‚Äî literal, case-sensitive personal identifiers, marked PRIVATE and **never committed here** (gitignored as defense-in-depth); read locally from `~/.config/pii-scan/scrub-rules.json` (override `$PII_RULES`) and in CI from the `PII_SCRUB_RULES` secret; (2) the **ignore patterns** `pii-ignore-patterns.txt` ‚Äî regexes for documented false-positive *shapes* (no PII, tracked). Plus a dotfiles-only third layer: (3) the **allowlist** `pii-allowlist.txt` ‚Äî intentionally-public values for *this* repo (your GitHub handle; generic vendor names like `Anthropic`/`OpenAI` that the shared denylist scrubs elsewhere but are intrinsic to a Claude Code dotfiles repo). A denylist hit clears only if `value in allowEntry and allowEntry in line` (whitelisting a full email also clears its domain substring, but only on lines that contain the full email) or the value matches an ignore regex. **Don't edit the shared denylist to silence a dotfiles false positive ‚Äî add to `pii-allowlist.txt` instead.** Two run modes: `--staged` (staged blobs via `git show :file`, used by the hook) and default (all `git ls-files`). **Fails open when the denylist is absent** (machines without the denylist still commit) unless `--require-rules` (CI) is passed, so a missing secret is loud. Wired two ways: the tracked **`.githooks/pre-commit`** (enabled by `install.sh` via `git config core.hooksPath .githooks`) runs `--staged` locally; the **`pii-scan` job in `.github/workflows/ci.yml`** runs `--require-rules` on push/PR to `main`, materializing the secret to a temp file. Bypass a local block once with `git commit --no-verify`.
- **`.github/workflows/ci.yml` ‚Äî the single-gate CI pattern.** shellcheck + gitleaks + pii-scan are three jobs in *one* workflow plus a final **`ci` job named `CI`** that `needs:` all of them. **Branch protection requires exactly one check: `CI`** ‚Äî uniform across every repo, so there is nothing per-repo to remember (`needs:` only works within a single file, which is why they were merged). The gate is **`if: always()`** (without it the gate is *skipped* when a dependency fails, so the required check never reports failure and the PR merges anyway) and loops over `needs.*.result`, treating **`skipped` as a pass** so the self-skipping `pii-scan` (fork/Dependabot PRs) does not block them. Add/remove jobs freely ‚Äî just keep them in the gate's `needs:` and keep the gate named `CI`. **`scorecard.yml` stays separate and is NOT in the gate:** it has no `pull_request` trigger and is public-gated, so it never produces a PR check (requiring it would wedge PRs at "Expected ‚Äî waiting for status").
- **`claude/settings.json.example`** ‚Äî the only tracked Claude settings file: the seed for the real copy at `~/.claude/settings.json`, carrying just the `SessionStart` ‚Üí `claude-stamp-tmux` hook (see its note above). The live settings file is **per-machine and untracked by design** ‚Äî Claude Code writes to it at runtime (`/model` saves, "always allow" permission rules, plugin toggles), the same reason `~/.claude.json` is unmanaged; the former tracked-and-symlinked author config was dropped (a `/model` change silently edited the repo) and `claude/settings.json` is now gitignored. MCP is *not* managed here: claude.ai connectors sync via the Anthropic account automatically, and local/stdio MCP servers live in the stateful, non-symlinked `~/.claude.json` (none today ‚Äî sync via a merge step if added, never commit `~/.claude.json`).
